# Agent 测试规格：gameplay-programmer

## Agent 概述
职责领域：游戏机制代码、玩家系统、战斗实现与交互功能。
不负责：UI 实现（ui-programmer）、AI 行为树（ai-programmer）、引擎/渲染系统（engine-programmer）。
模型层级：Sonnet（默认）。
未分配关卡 ID。

---

## 静态断言（结构检查）

- [ ] `description:` 字段存在且领域明确（引用游戏机制 / 玩家系统）
- [ ] `allowed-tools:` 列表包含 Read、Write、Edit、Bash、Glob、Grep——不含仅编排类 Agent 需要的工具
- [ ] 模型层级为 Sonnet（专员的默认层级）
- [ ] Agent 定义未主张对 UI、AI 行为或引擎/渲染代码拥有权

---

## 测试用例

### 用例 1：领域内请求——合适的输出
**输入**："实现近战连击系统：三次连续普攻可衔接为终结技。"
**预期行为**：
- 按项目语言（GDScript/C#）和编码标准产出代码或代码骨架
- 将连击状态追踪、输入窗口计时、终结技触发逻辑定义为独立的、可测试的方法
- 若上下文中提供了对应的 GDD 章节，则引用该章节
- 不实现 UI 反馈（委托给 ui-programmer）或 AI 反应（委托给 ai-programmer）
- 按编码标准，在所有公开方法上包含文档注释

### 用例 2：领域外请求——正确重定向
**输入**："构建带暂停和设置面板的主菜单界面。"
**预期行为**：
- 不产出菜单实现代码
- 明确声明这超出其职责范围
- 将请求重定向给 `ui-programmer`
- 可注明若暂停菜单需要读取游戏状态，可提供状态 API 接口

### 用例 3：领域边界——线程标记
**输入**："连击系统造成帧卡顿；能否加入多线程来分摊输入处理？"
**预期行为**：
- 不单方面实现多线程或异步系统
- 向 `engine-programmer` 标记线程问题，并清晰描述热路径
- 可产出非多线程的重构方案，作为降低每帧工作量的安全过渡步骤
- 记录上报情况，以便 lead-programmer 知晓

### 用例 4：与已批准 ADR 冲突
**输入**："将伤害计算改为直接使用浮点数累加，而不是 ADR-003 中的定点数公式。"
**预期行为**：
- 识别提议的变更违反了 ADR-003（已批准状态）
- 不静默实现该违规
- 向 `lead-programmer` 标记冲突，附 ADR 引用和权衡说明
- 仅在 lead-programmer 或 technical-director 明确作出覆盖决策后才实现

### 用例 5：上下文传递——按 GDD 规格实现
**输入**：上下文中提供了"PlayerCombat"的 GDD。请求："实现战斗 GDD 中的体力消耗公式。"
**预期行为**：
- 读取所提供 GDD 的公式章节
- 按原样实现公式——不擅自引入新变量或调整系数
- 将体力消耗设为数据驱动值（外部配置），而非硬编码常量
- 注明 GDD 边缘情况章节中的所有边缘情况，并在代码中处理

## 协议合规

- [ ] 保持在声明领域内（机制、玩家系统、战斗）
- [ ] 将领域外请求重定向给正确的 Agent（ui-programmer、ai-programmer、engine-programmer）
- [ ] 返回结构化产出（代码骨架、方法签名、内联注释），而非自由形式意见
- [ ] 未经明确授权，不修改 `src/gameplay/` 或 `src/core/` 之外的文件
- [ ] 标记 ADR 违规，而非静默覆盖
- [ ] 将游戏数值设为数据驱动，永不硬编码

---

## 覆盖说明
- 连击系统测试（用例 1）应通过 `tests/unit/gameplay/` 中的单元测试验证
- 线程上报（用例 3）验证 Agent 不越界进入引擎领域
- ADR 冲突（用例 4）确认 Agent 遵守架构治理流程
- 用例 1 和用例 5 共同验证 Agent 按规格实现，而非自行发挥
