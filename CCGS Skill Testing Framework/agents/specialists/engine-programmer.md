# Agent 测试规格：engine-programmer

## Agent 概述
职责领域：渲染流水线、物理集成、内存管理、资源加载与核心引擎框架。
不负责：游戏玩法机制（gameplay-programmer）、编辑器/调试工具 UI（tools-programmer）。
模型层级：Sonnet（默认）。
未分配关卡 ID。

---

## 静态断言（结构检查）

- [ ] `description:` 字段存在且领域明确（引用渲染 / 内存 / 引擎核心）
- [ ] `allowed-tools:` 列表包含 Read、Write、Edit、Bash、Glob、Grep
- [ ] 模型层级为 Sonnet（专员的默认层级）
- [ ] Agent 定义未主张对游戏玩法机制或工具 UI 拥有权

## 测试用例

### 用例 1：领域内请求——合适的输出
**输入**："为弹药实现自定义对象池，避免每帧分配。"
**预期行为**：
- 产出带 acquire/release 接口的引擎级对象池实现
- 对象池类型化为弹药对象类型，使用预分配的固定大小存储
- 提供线程安全说明（或清晰标注为单线程模式并说明原因）
- 按编码标准在公开 API 上包含文档注释
- 输出与项目配置的引擎和语言兼容

### 用例 2：领域外请求——正确重定向
**输入**："添加一个带音量滑块和'返回主菜单'按钮的暂停菜单界面。"
**预期行为**：
- 不产出 UI 界面代码
- 明确声明菜单界面属于 `ui-programmer` 的职责范围
- 将请求重定向给 `ui-programmer`
- 可注明可为 ui-programmer 提供引擎级音频音量 API 端点

### 用例 3：内存泄漏诊断
**输入**："每次关卡加载时内存增加约 50MB，且从不释放。我们怀疑是资源加载系统的问题。"
**预期行为**：
- 产出系统性诊断方案：引用计数审计、资源句柄生命周期检查、缓存失效审查
- 识别可能原因（孤立资源句柄、循环引用、从不驱逐的缓存）
- 针对识别出的泄漏模式产出具体修复方案
- 提供验证修复效果的测试（加载前内存基线，卸载后测量，确认恢复到基线）

### 用例 4：跨领域协调——共享系统优化
**输入**："需要优化物理宽相，但游戏玩法系统与物理查询 API 紧密耦合。"
**预期行为**：
- 不单方面变更物理查询 API 接口（会破坏 gameplay-programmer 的代码）
- 与 `lead-programmer` 协调，安全地规划变更
- 提出迁移路径：新优化 API 与旧 API 并存，设置废弃过渡期
- 在继续之前记录协调需求

### 用例 5：上下文传递——检查引擎版本参考
**输入**：上下文中提供了引擎版本参考（Godot 4.6）。请求："为项目设置默认物理引擎。"
**预期行为**：
- 读取引擎版本参考，注意 Godot 4.6 的变化：Jolt 物理现为默认引擎
- 产出考虑到"Jolt 为默认"变更的配置指南（4.6 迁移说明）
- 标记 GodotPhysics 与 Jolt 之间可能影响现有代码的 API 差异
- 不在未注明适用于旧版本的情况下，建议已废弃或 4.6 之前的物理设置步骤

## 协议合规

- [ ] 保持在声明领域内（渲染、物理、内存、资源加载、核心框架）
- [ ] 将 UI/菜单请求重定向给 ui-programmer
- [ ] 返回结构化产出（实现代码、诊断步骤、迁移方案）
- [ ] 变更共享 API 接口前与 lead-programmer 协调
- [ ] 建议引擎特定 API 前检查引擎版本参考
- [ ] 为修复提供测试证据（内存修复前/后、性能测量）
