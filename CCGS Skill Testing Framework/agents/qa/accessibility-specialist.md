# Agent 测试规格：accessibility-specialist

## Agent 摘要
**领域：** 输入重映射、文字缩放、色盲模式、屏幕阅读器支持以及无障碍标准合规性（WCAG、平台认证）。
**不拥有：** 整体 UX 流程设计（ux-designer）、视觉美术风格方向（art-director）。
**模型层级：** Sonnet（默认）。
**关卡 ID：** 无。

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且针对特定领域（引用无障碍/包容性设计/WCAG）
- [ ] `allowed-tools:` 列表包含 Read、Write、Edit、Bash、Glob、Grep
- [ ] 模型层级为 Sonnet（专家类默认值）
- [ ] Agent 定义不声明对 UX 流程或视觉美术风格的权限

---

## 测试用例

### 用例 1：领域内请求 — 适当的输出
**输入：** "请评审玩家 HUD 的无障碍性。"
**预期行为：**
- 对 HUD 规格或截图进行审计，检查：
  - 对比度比率（标记任何 AA 低于 4.5:1 或 AAA 低于 7:1 的文字）
  - 颜色编码信息的替代表现形式（例如：敌方血条仅使用颜色，无形状区分）
  - 文字大小（标记在 1080p 下任何低于 16px 等效大小的文字）
  - 关键状态元素的屏幕阅读器或 TTS 注释可用性
- 产出带有具体元素名称及其不符合标准项的优先级发现列表
- 不重新设计 HUD — 仅产出供 ux-designer 和 ui-programmer 处理的发现

### 用例 2：领域外请求 — 正确重定向
**输入：** "请设计整体游戏流程：主菜单 → 角色选择 → 加载 → 游戏中 → 暂停 → 结算。"
**预期行为：**
- 不产出 UX 流程架构
- 明确说明整体游戏流程设计属于 `ux-designer`
- 将请求重定向至 `ux-designer`
- 可以说明在流程设计完成后，可以评审该流程的无障碍顾虑（例如时间限制、认知负荷）

### 用例 3：色盲模式冲突
**输入：** "针对绿色盲提出的色盲模式将敌方红色血条替换为橙色，但美术调色板已将橙色用于友方单位。"
**预期行为：**
- 识别冲突：色盲模式与已建立的友方单位调色板之间的橙色碰撞
- 不单方面更改美术调色板（该权限属于 art-director）
- 向 `art-director` 标记冲突，并描述具体的视觉重叠情况
- 提出不需要更改调色板的替代区分策略（例如形状/图标叠加、图案填充、图形符号）

### 用例 4：无障碍功能的 UI 状态需求
**输入：** "库存的屏幕阅读器支持需要系统将物品名称和数量作为无障碍文本节点暴露出来。"
**预期行为：**
- 产出无障碍需求规格，定义每个库存元素所需的无障碍文本属性
- 识别实现无障碍文本节点需要 UI 系统的变更
- 协调 `ui-programmer` 实施所需的无障碍文本节点暴露
- 不自行实现 UI 系统变更

### 用例 5：上下文传递 — WCAG 2.1 目标
**输入上下文：** 项目无障碍目标：WCAG 2.1 AA 合规。请求："请评审对话系统的无障碍性。"
**预期行为：**
- 引用与对话相关的具体 WCAG 2.1 AA 成功标准（例如：1.4.3 对比度最低要求、1.4.4 文字缩放、2.2.1 时间可调整 — 用于自动推进的对话）
- 使用标准中的精确准则编号和名称，而非概括性描述
- 每条发现附带其不符合的具体准则
- 注明哪些准则超出 AA 范围（仅 AAA），以避免被错误标记为失败

## 协议合规性

- [ ] 严守声明的领域（重映射、文字缩放、色盲模式、屏幕阅读器、标准合规性）
- [ ] 将 UX 流程设计重定向至 ux-designer，美术调色板决策重定向至 art-director
- [ ] 返回带有具体元素名称、对比度比率和准则引用的结构化发现
- [ ] 不实现 UI 变更 — 协调 ui-programmer 进行实现
- [ ] 在提供合规目标时按编号引用具体的 WCAG 准则
- [ ] 将无障碍需求与美术决策的冲突标记给 art-director

---

## 覆盖说明
- HUD 审计（用例 1）应产出可作为无障碍故事追踪到冲刺待办列表的发现
- 色盲冲突（用例 3）确认 agent 尊重 art-director 对调色板的权限
- WCAG 准则（用例 5）验证 agent 精确使用标准，而非泛泛而谈
