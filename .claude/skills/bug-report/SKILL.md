# Bug 报告

Bug 报告技能以标准化格式创建、分析和更新 Bug，确保报告中包含足够的信息以便高效修复。

---

## 模式选择

根据传入的参数选择运行模式：

| 参数 | 模式 | 用途 |
|------|------|------|
| `describe [描述]` | **描述模式** | 从用户口头描述创建结构化 Bug 报告 |
| `analyze [文件路径]` | **分析模式** | 分析代码文件，识别潜在 Bug |
| `verify [bug-id]` | **验证模式** | 验证 Bug 是否已在当前代码中修复 |
| `close [bug-id]` | **关闭模式** | 追加关闭记录，标记 Bug 为已解决 |

若未提供参数：使用**描述模式**，等待用户描述 Bug。

---

## 描述模式

**使用场景**：用户报告了一个 Bug，需要将其转化为标准化 Bug 报告。

### 步骤 1：信息提取

从用户描述中提取（若描述不完整，提出问题补全）：

**必需信息：**
- 发生了什么？（实际行为）
- 期望发生什么？（预期行为）
- 如何触发？（复现步骤）
- 发生的频率？（每次 / 有时 / 偶尔）

**可选但有用的信息：**
- 平台/环境（PC / Mobile / Web / 特定操作系统）
- 游戏版本或 commit hash
- 是否有截图或日志

**自动推断（从描述中）：**
- 受影响系统（战斗 / UI / 音频 / 保存等）
- 严重性（参见下方严重性指南）

**严重性分类指南：**

| 级别 | 标准 |
|------|------|
| **S1 - Critical** | 游戏崩溃、数据丢失、进度无法继续、在主关键路径上 100% 复现 |
| **S2 - High** | 核心功能损坏（但游戏未崩溃）、影响主要游戏系统、高频复现 |
| **S3 - Medium** | 功能降级但有绕过方式、影响次要系统、偶发复现 |
| **S4 - Low** | 外观问题、轻微的不一致、极少复现或仅在边缘场景触发 |

若用户使用了 `--severity` 参数，直接使用该严重性，不自动推断。

### 步骤 2：生成 Bug ID

读取 `production/qa/bugs/` 目录，Glob `*.md` 文件，提取已使用的最大 Bug 编号。新 Bug ID 为最大编号 + 1，格式：`BUG-[4位数字]`，例如 `BUG-0043`。

若目录不存在，从 `BUG-0001` 开始。

### 步骤 3：生成报告并写入

写入 `production/qa/bugs/[BUG-ID]-[简短描述-kebab-case].md`：

```markdown
# [BUG-ID] — [简短描述（10字以内）]

> **严重性**：[S1-Critical / S2-High / S3-Medium / S4-Low]
> **优先级**：[P1 / P2 / P3 / P4]（在分诊时分配，初始留空或由报告者建议）
> **状态**：Open
> **受影响系统**：[系统名称]
> **报告日期**：[日期]
> **报告者**：[用户 / 自动分析]

---

## 描述

[1-2 句话概述 Bug——"在 [条件] 下，[行为] 而非 [预期行为]。"]

## 复现步骤

1. [精确步骤 1]
2. [精确步骤 2]
3. [精确步骤 3（若需要）]

**复现频率**：[每次 / 高频（>50%）/ 偶发（10-50%）/ 罕见（<10%）]

## 预期行为

[玩家/用户应该看到什么]

## 实际行为

[玩家/用户实际看到了什么]

## 环境

| 字段 | 值 |
|------|----|
| 平台 | [PC / Mobile / Web / 主机] |
| 操作系统 | [若已知] |
| 游戏版本 | [版本号 / commit hash / 未知] |
| 引擎版本 | [若已知] |

## 附加上下文

[相关日志、截图链接、相关代码区域等。若无则写"无"。]

## 根本原因假设

[若有明显迹象，写出对代码层面根本原因的假设。若不明确，写"待调查"。]

---

## 变更历史

| 日期 | 动作 | 备注 |
|------|------|------|
| [日期] | 已创建 | 初始报告 |
```

报告写入后，在对话中确认：
> "Bug 报告已创建：`[BUG-ID]` — [简短描述]
> 路径：`production/qa/bugs/[BUG-ID]-[描述].md`
> 严重性：[S级别] | 状态：Open"

## 分析模式

**使用场景**：分析一个代码文件，主动识别潜在 Bug。

### 步骤 1：读取目标文件

读取指定文件路径的完整内容。

同时读取（若存在）：
- 相关 GDD（Grep 文件中引用的系统名称）
- 同一目录下的测试文件

### 步骤 2：Bug 识别扫描

对文件进行以下维度的分析：

**空指针/空引用风险：**
- 在没有空值检查的情况下使用的变量
- 链式调用但未验证中间值
- 可能为 null 的函数返回值直接使用

**边界条件：**
- 数组/列表访问未检查索引范围
- 除以可能为零的变量
- 整数溢出风险（超大数值计算）

**状态机错误：**
- 未处理的状态转换
- 遗漏的默认情况（switch/match 无 default）
- 可以同时激活的互斥状态

**资源泄漏：**
- 未释放的连接或文件句柄
- 未注销的信号/事件监听器
- 场景卸载时未清理的节点引用

**竞态条件（异步代码）：**
- 共享状态在 await/async 前后的修改
- 不正确的协程时序假设

**逻辑错误：**
- 永远为真或永远为假的条件
- 循环中的 off-by-one 错误
- 变量遮蔽（内部作用域重用了外部变量名）

### 步骤 3：对每个发现生成 Bug 报告

对每个识别到的问题，按描述模式写入一份完整的 Bug 报告，并注明：
- **报告者**：自动分析
- **复现频率**：根据代码路径估计（仅在特定条件下触发的问题标为"偶发"）
- **根本原因假设**：直接指出有问题的代码区域和行号

输出分析摘要：
> "在 `[文件路径]` 中发现 [N] 个潜在 Bug：
> - S1-Critical：[N] 个
> - S2-High：[N] 个
> - S3-Medium：[N] 个
> - S4-Low：[N] 个
>
> 已写入 [N] 份 Bug 报告。"

## 验证模式

**使用场景**：确认一个已报告的 Bug 在当前代码状态下是否已修复。

### 步骤 1：读取 Bug 报告

读取 `production/qa/bugs/[BUG-ID]-*.md`（Glob 匹配）。

提取：受影响系统、根本原因假设、复现步骤。

### 步骤 2：在代码中定位相关区域

基于受影响系统和根本原因假设，Grep 相关代码：

```
Grep pattern="[系统名称 / 相关函数名]" glob="src/**/*.{gd,cs,cpp}"
```

读取相关文件的相关部分。

### 步骤 3：评估修复状态

对照 Bug 报告中的根本原因假设评估当前代码：

- **VERIFIED FIXED**：代码中的问题已不存在，或已有明确修复
- **STILL PRESENT**：问题仍然存在
- **CANNOT VERIFY**：代码已发生变化，但无法确定 Bug 是否已修复（需要运行时测试）

### 步骤 4：更新 Bug 报告

在 Bug 报告的变更历史中追加：

```markdown
| [日期] | 验证 | [VERIFIED FIXED / STILL PRESENT / CANNOT VERIFY] — [简短说明] |
```

若 VERIFIED FIXED，将状态更新为 `Verified Fixed`，但**不要关闭**——关闭需要通过关闭模式（`close`）进行。

---

## 关闭模式

**使用场景**：记录 Bug 修复已通过测试，正式关闭。

### 步骤 1：读取 Bug 报告

读取 `production/qa/bugs/[BUG-ID]-*.md`。

### 步骤 2：检查前置条件

- 状态是否为 `Verified Fixed` 或 `Open`？
- 若状态为 `Open`（未验证），询问："此 Bug 尚未验证修复，是否确认直接关闭？"
  - `[A] 是——强制关闭` / `[B] 否——先运行 verify 模式`

### 步骤 3：追加关闭记录

在 Bug 文件中追加关闭记录：

```markdown
## Closure Record
**Closed**: [日期]
**Resolution**: Fixed — [一句话描述修改内容]
**Fix commit / PR**: [若已知]
**Verified by**: qa-tester
**Closed by**: [用户]
**Regression test**: [测试文件路径，或"手动验证"]
**Status**: Closed
```

将文件顶部的 `**Status**: Open` 字段更新为 `**Status**: Closed`。

询问："是否可以更新 `production/qa/bugs/[BUG-ID].md` 将其标记为已关闭？"

关闭后，检查 `production/qa/bug-triage-*.md`——若 Bug 出现在未完结的分诊报告中，提示："Bug [ID] 已在分诊报告中引用。运行 `/bug-triage` 以刷新未解 Bug 数量。"

确认：
> "BUG-[ID] 已关闭。
> 状态：Closed | 关闭日期：[日期]"

---

## 阶段 3：保存报告

向用户展示完成的 Bug 报告。

询问："是否可以将报告写入 `production/qa/bugs/BUG-[NNNN].md`？"

若是：写入文件，如目录不存在则创建。判定：**COMPLETE** — Bug 报告已提交。

若否：停止。判定：**BLOCKED** — 用户拒绝写入。

---

## 阶段 4：下一步

保存后，根据模式提出建议：

**提交报告后（描述/分析模式）：**
- 运行 `/bug-triage` 对现有未解 Bug 进行优先级排序
- 若为 S1 或 S2：运行 `/hotfix [BUG-ID]` 启动紧急修复工作流

**Bug 修复后（开发者确认修复已提交）：**
- 运行 `/bug-report verify [BUG-ID]` — 在关闭前确认修复确实有效
- 未经验证不得将 Bug 标记为已关闭——未验证的修复仍为 Open 状态

**验证返回 VERIFIED FIXED 后：**
- 运行 `/bug-report close [BUG-ID]` — 写入关闭记录并更新状态
- 运行 `/bug-triage` 刷新未解 Bug 数量，将其从活跃列表中移除
