# 不稳定测试检测

不稳定测试是指在没有代码变更的情况下有时通过、有时失败的测试。
不稳定测试在某种程度上比没有测试更糟——它们训练团队忽视 CI 红灯，从而掩盖真正的失败。
本技能识别它们，解释可能的原因，并推荐是隔离还是修复每个测试。

**输出：** 更新的 `tests/regression-suite.md` 隔离部分 + 可选的 `production/qa/flakiness-report-[date].md`

**何时运行：**
- 打磨阶段（测试已经过多次运行；统计信号可靠）
- 当开发者开始将 CI 失败归因于"可能是不稳定"时
- `/regression-suite` 识别出需要诊断的已隔离测试后

---

## 1. 解析参数

**模式：**
- `/test-flakiness [ci-log-path]` — 分析特定的 CI 运行日志文件
- `/test-flakiness scan` — 扫描 `.github/` 或标准日志输出目录中的所有可用 CI 日志
- `/test-flakiness registry` — 读取现有 regression-suite.md 的隔离部分并提供针对已知不稳定测试的修复指导
- 无参数 — 自动检测：若 CI 日志可访问则运行 `scan`，否则运行 `registry`

---

## 2. 定位 CI 日志数据

### 方式 A — GitHub Actions（推荐）

检查测试结果制品：
```bash
ls -t .github/ 2>/dev/null
ls -t test-results/ 2>/dev/null
```

Godot 项目：GdUnit4 输出兼容 JUnit 格式的 XML 结果，检查 `test-results/` 中的 `.xml` 文件。

Unity 项目：game-ci 测试运行器默认将 NUnit XML 输出到 `test-results/`。

Unreal 项目：自动化日志写入 `Saved/Logs/`，Grep `Result: Success` 和 `Result: Fail` 模式。

### 方式 B — 本地日志文件

若提供了路径参数，直接读取该文件。

### 方式 C — 无日志数据

若未找到日志：
> "未找到 CI 日志数据。要检测不稳定测试，本技能需要来自多次运行的测试结果历史。选项：
> 1. 至少运行测试套件 3 次并收集输出日志
> 2. 检查 CI 流水线输出并将日志保存到 `test-results/`
> 3. 运行 `/test-flakiness registry` 查看已在 `tests/regression-suite.md` 中标记为不稳定的测试"

停止并询问用户选择哪个选项。

## 3. 解析测试结果

对找到的每个 CI 日志或结果文件，解析：

**JUnit XML 格式**（GdUnit4 / Unity）：
- Grep `<testcase name=` 获取测试名称
- Grep `<failure` 或 `<error` 识别失败
- 解析 `classname` 和 `name` 属性获取完整测试标识符

**纯文本日志**：
- Grep 通过/失败模式：
  - Godot：紧邻测试名称的 `PASSED` / `FAILED`
  - Unreal：`Result: Success` / `Result: Fail`
  - Unity：`Test passed` / `Test failed`

构建表格：`test_id → [run1_result, run2_result, run3_result, ...]`

---

## 4. 识别不稳定测试

若某个测试在结果历史中出现了 PASS 和 FAIL 两种结果，且两次运行之间没有代码变更，则该测试是**不稳定的**。

不稳定性阈值：
- **高不稳定性**：超过 25% 的运行失败——立即隔离
- **中等不稳定性**：5-25% 的运行失败——尽快调查并修复
- **低/疑似不稳定性**：1-5% 的运行失败——监控；可能是真正的罕见失败

对每个不稳定测试，分类可能原因：

### 原因分类

| 原因 | 症状 | 修复方向 |
|------|------|---------|
| **时序/异步** | 在等待信号或计时器后失败；通过率与系统负载相关 | 添加明确的 await/同步；避免基于时间的延迟 |
| **顺序依赖** | 在特定其他测试后运行时失败；单独运行时通过 | 添加适当的 setup/teardown；确保测试隔离 |
| **随机种子** | 无规律地间歇性失败；涉及 RNG | 传入明确的种子；测试中不使用 `randf()` |
| **资源泄漏** | 在测试运行后期失败更频繁 | 在 teardown 中修复清理；检查孤立节点（Godot）或对象处置（Unity） |
| **外部状态** | 当前一个测试留下文件、场景或全局状态时失败 | 将测试与文件系统隔离；使用内存模拟 |
| **浮点数** | 在 `== 0.5` 等比较上失败 | 使用近似比较（`is_equal_approx`、`Assert.AreApproximately`） |
| **场景/预制体加载竞争** | 场景尚未就绪时失败 | 实例化后等待一帧；使用 `await get_tree().process_frame` |

使用 Grep 检查测试文件中的时序调用、randf、全局状态访问或浮点数等值比较，以缩小原因范围。

## 5. 推荐操作

对每个不稳定测试：

**隔离（高不稳定性）：**
> "立即隔离此测试。在 CI 中通过添加 `@pytest.mark.skip` / `[Ignore]` / `GdUnitSkip` 注解禁用它。在 `tests/regression-suite.md` 隔离部分记录它。测试现在仅为选用。在移除隔离前先修复根本原因。"

**尽快调查并修复（中等不稳定性）：**
> "此测试间歇性不可靠。根本原因似乎是 [原因]。建议修复：[基于原因分类的具体修复]。暂不隔离——直接修复测试。"

**监控（低/疑似不稳定性）：**
> "此测试显示疑似不稳定性。在隔离前收集更多运行数据。在回归套件中将其记为'疑似'。"

## 6. 生成报告

### 对话内摘要

```
## 不稳定性检测结果

**分析的运行次数**：[N]
**追踪的测试数**：[N]

### 发现的不稳定测试

| 测试 | 系统 | 失败率 | 可能原因 | 推荐操作 |
|------|------|--------|---------|---------|
| [test_name] | [system] | [N]% | 时序 | 隔离 + 修复异步 |
| [test_name] | [system] | [N]% | 浮点比较 | 修复：使用近似比较 |
| [test_name] | [system] | [N]% | 顺序依赖 | 调查 teardown |

### 干净测试（未检测到不稳定性）

[N] 个测试在 [N] 次运行中结果一致——未检测到不稳定性。

### 数据局限性

[若可用运行次数少于 5 次，请注明——运行次数越少，统计置信度越低]
```

## 7. 更新回归套件 + 可选报告文件

询问："我可以将发现的不稳定测试更新到 `tests/regression-suite.md` 的隔离部分吗？"

若是：使用 `Edit` 在隔离测试表格中追加条目。
永远不要删除现有隔离条目——只添加新条目。

单独询问："我可以将完整的不稳定性报告写入 `production/qa/flakiness-report-[date].md` 吗？"

完整报告包括每个测试的原因分析和引擎特定修复片段。

写入后：

- 对每个已隔离的测试："添加引擎特定的跳过注解以在 CI 中禁用此测试。修复根本原因后重新启用。"
- 对可修复的测试："[test] 的修复很简单——将第 [N] 行的等值比较改为使用 `is_equal_approx`。"
- 摘要："所有隔离注解应用后，CI 应绿灯运行。在发布门控前为 [N] 个已隔离测试安排修复工作。"

---

## 协作协议

- **永远不要删除测试文件** — 隔离意味着注解 + 列表，不是删除
- **统计置信度很重要** — 运行次数少于 3 次时，将发现标记为"疑似"而非"已确认"；询问是否有更多运行数据
- **修复始终是目标** — 隔离是临时的；即使推荐隔离，也要指出修复方向
- **写入前询问** — 回归套件更新和报告文件都需要明确批准。写入时：结论：**COMPLETE** — 不稳定性报告已写入。拒绝时：结论：**BLOCKED** — 用户拒绝写入。
- **CI 中的不稳定性是团队问题** — 清晰呈现列表和推荐操作；不要在团队不知情的情况下悄悄隔离
