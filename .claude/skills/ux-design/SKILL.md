当本技能被调用时：

## 1. 解析参数并确定模式

根据参数存在三种编写模式：

| 参数 | 模式 | 输出文件 |
|------|------|----------|
| `hud` | HUD 设计 | `design/ux/hud.md` |
| `patterns` | 交互模式库 | `design/ux/interaction-patterns.md` |
| 任意其他值（如 `main-menu`、`inventory`） | 屏幕或流程的 UX 规格 | `design/ux/[参数].md` |
| 无参数 | 询问用户 | （见下文） |

**若未提供参数**，不要失败——改为询问。使用 `AskUserQuestion`：
- "我们今天要设计什么？"
  - 选项："某个特定屏幕或流程（我来命名）"、"游戏 HUD"、"交互模式库"、"我不确定——帮我弄清楚"

若用户选择"我来命名"或直接输入屏幕名称，将其转换为 kebab-case 格式作为文件名
（例如"主菜单"对应 `main-menu`）。

---

## 2. 收集上下文（读取阶段）

在询问用户任何事情之前，先读取所有相关上下文。
本技能的价值在于带着充分的信息到达。

### 2a：必读内容

- **游戏概念**：读取 `design/gdd/game-concept.md`——若缺失，给出警告：
  > "未找到游戏概念。请先运行 `/brainstorm` 建立游戏基础，然后再进行 UX 设计。"
  > 若用户要求则继续。

### 2b：玩家旅程

若 `design/player-journey.md` 存在则读取。对于每个相关部分，提取：
- 本屏幕出现在哪些旅程阶段？
- 玩家到达本屏幕时的情感状态是什么？
- 本屏幕在旅程中服务于哪个玩家需求？
- 本屏幕承载了哪些旅程关键时刻？

若玩家旅程文件不存在，记录缺口并继续：
> "在 `design/player-journey.md` 未找到玩家旅程地图。没有它进行设计
> 意味着我们将对玩家情境做出假设。考虑在完成本规格草稿后再进行玩家旅程会话。"

### 2c：GDD UI 需求

Glob `design/gdd/*.md` 并 grep `UI Requirements` 节。读取 UI 需求部分
按名称或类别引用了本屏幕的所有 GDD。

这些 GDD UI 需求是本规格的**需求输入**。将它们收集为本规格必须满足的约束列表。

若正在设计 HUD，读取所有 GDD 的 UI 需求部分——HUD 聚合了所有系统的需求。

### 2d：现有 UX 规格

Glob `design/ux/*.md`，记录哪些屏幕已有规格。对于将与当前屏幕相互链接的屏幕，
读取它们的导航/流程部分，找到本规格需要匹配的入口和出口。

### 2e：交互模式库

若 `design/ux/interaction-patterns.md` 存在，仅读取模式目录索引（模式名称及其单行描述），
不读取完整的模式细节。这告诉你哪些模式已存在，以便引用而非重新发明。

### 2f：美术圣经

检查 `design/art/art-bible.md`。若找到，读取视觉方向部分。
UX 布局必须与已承诺的美学方向保持一致。

### 2g：无障碍需求

检查 `design/accessibility-requirements.md`。若找到则读取。
本规格必须满足其中承诺的无障碍等级。

### 2h：输入方式（来自项目配置）

读取 `.claude/docs/technical-preferences.md` 并提取 `## Input & Platform` 部分。
在整个技能中存储这些值——它们驱动交互地图并告知无障碍需求：

- **Input Methods**（输入方式）— 如键盘/鼠标、手柄、触控、混合
- **Primary Input**（主要输入）— 本游戏的主导输入方式
- **Gamepad Support**（手柄支持）— Full / Partial / None
- **Touch Support**（触控支持）— Full / Partial / None
- **Target Platforms**（目标平台）— 用于安全区域和宽高比决策

若该部分未配置（`[TO BE CONFIGURED]`），询问一次：
> "输入方式尚未配置。这款游戏目标什么？"
> 选项："仅键盘/鼠标"、"仅手柄"、"两者（PC + 主机）"、"触控（移动）"、"以上全部"
>
>（运行 `/setup-engine` 可永久保存此设置，下次不再询问。）

存储答案供本次会话使用。**不要**在每个部分或每个屏幕再次询问。

### 2i：呈现上下文摘要

在任何设计工作开始之前，向用户呈现简要摘要：

> **正在设计：[屏幕/流程名称]**
> - 模式：[UX 规格 / HUD 设计 / 模式库]
> - 旅程阶段：[来自 player-journey.md，或"未知——无旅程地图"]
> - 输入本规格的 GDD 需求：[数量和名称，或"未找到"]
> - 已有规格的相关屏幕：[列表，或"暂无"]
> - 已有的已知模式：[数量，或"暂无模式库"]
> - 无障碍等级：[来自需求文档，或"尚未定义"]
> - 输入方式：[来自 technical-preferences.md，或"已在上方询问"]

然后询问："还有什么需要我在开始前阅读的吗？还是直接开始？"

## 2b. 改造模式检测

在创建骨架之前，检查目标输出文件是否已存在。

Glob `design/ux/[filename].md`（其中 `[filename]` 是第 1 阶段确定的输出路径）。

**若文件已存在——改造模式：**
- 完整读取文件
- 对于每个预期的部分，检查正文是否有真实内容（超出 `[待设计]` 占位符）还是空的/占位符
- 向用户呈现部分状态摘要：

> "在 `design/ux/[filename].md` 找到现有 UX 规格。以下是已完成的内容：
>
> | 部分 | 状态 |
> |------|------|
> | 概述与上下文 | [Complete / Empty / Placeholder] |
> | 玩家旅程整合 | ... |
> | 屏幕布局与信息架构 | ... |
> | 交互模型 | ... |
> | 反馈与状态通信 | ... |
> | 无障碍 | ... |
> | 边界情况与错误状态 | ... |
> | 待解问题 | ... |
>
> 我将仅处理 [N] 个未完成的部分——现有内容不会被覆盖。"

- 跳过第 3 节（骨架创建）——文件已存在
- 在第 4 阶段（部分编写）中，仅处理状态为 Empty 或 Placeholder 的部分
- 使用 `Edit` 原地填充占位符，而不是创建新骨架

**若文件不存在——全新编写模式：**
正常进入第 3 阶段（创建文件骨架）。

## 4. 逐节编写

按顺序遍历每个部分。**对于每个部分**，遵循以下循环：

```
上下文 -> 问题 -> 选项 -> 决策 -> 草稿 -> 审批 -> 写入
```

1. **上下文**：说明本部分需要包含的内容，并从第 2 阶段收集的上下文中提取相关约束。
2. **问题**：询问起草本部分所需的内容。对于有约束的选择使用 `AskUserQuestion`，
   对于开放式探索使用对话文本。
3. **选项**：存在设计选择时，呈现 2-4 种方案及其优缺点。在对话中解释推理，
   然后使用 `AskUserQuestion` 记录决策。
4. **决策**：用户选择一种方案或提供自定义方向。
5. **草稿**：在对话中撰写部分内容供审阅。明确标记临时假设。
6. **审批**："这是否准确体现了您的意图？写入文件前是否有修改？"
7. **写入**：使用 `Edit` 将 `[待设计]` 占位符替换为已审批的内容。确认写入。

写入每个部分后，更新 `production/session-state/active.md`。

### 部分指导：UX 规格模式

#### A 部分：目的与玩家需求

本部分是基础。其他所有决策均从此出发。

**需要询问的问题**：
- "本屏幕服务于哪个玩家目标？玩家在此想要**做什么**？"
- "若本屏幕不存在或难以使用，会出现什么问题？"
- "补全这句话：'玩家到达本屏幕时想要 ___。'"

与第 2 阶段收集的玩家旅程上下文进行交叉参考。
所述目的必须与旅程阶段和情感状态一致。

---

#### B 部分：到达时的玩家情境

**需要询问的问题**：
- "玩家第一次在游戏中遇到此屏幕是什么时候？"
- "到达此屏幕之前他们刚刚在做什么？"
- "设计应假设什么情感状态？（平静、紧张、好奇、时间紧迫）"
- "玩家主动到达此屏幕，还是被游戏引导到此处？"

若玩家旅程文档存在，可提供将此映射到旅程阶段的服务。

---

#### B2 部分：导航位置

本屏幕在游戏导航层级中处于什么位置？这是一段方向性叙述——不是完整的流程图。

**需要询问的问题**：
- "此屏幕从主菜单、暂停菜单、游戏内还是另一个屏幕进入？"
- "它是顶级目标（始终可访问）还是情境相关目标（仅在特定状态下可访问）？"
- "玩家能从游戏中多个位置进入此屏幕吗？"

呈现格式："此屏幕位于：[根] → [父级] → [本屏幕]" 加上所有备用入口路径。

---

#### B3 部分：入口与出口

映射玩家到达和离开此屏幕的所有方式。

**需要询问的问题**：
- "玩家到达此屏幕的所有方式是什么？"（列出每个触发器：按键、游戏事件、从其他屏幕跳转等）
- "玩家可以如何退出？退出后会发生什么？"（返回键、确认操作、超时、游戏事件）
- "是否有单向出口——玩家无法不重新开始就返回此屏幕？"

呈现为两个表格：

| 入口来源 | 触发器 | 玩家携带的情境 |
|----------|--------|----------------|
| [屏幕/事件] | [如何] | [到达时携带的状态/数据] |

| 出口目标 | 触发器 | 备注 |
|----------|--------|------|
| [屏幕/事件] | [如何] | [任何不可逆的状态变化] |

---

#### C 部分：布局规格

这是最大且最具交互性的部分。按子部分逐步进行：

**子部分 1——信息层级**（在任何布局之前建立）：
- 请用户列出此屏幕必须传达的每条信息。
- 然后请他们排序："玩家首先需要看到的最重要的一件事是什么？第二呢？什么可以被发现而不是立即可见？"
- 在转向区域之前呈现结果层级供审批。

**子部分 2——布局区域**：
- 根据信息层级，提议粗略的屏幕区域（页眉、内容区域、操作栏、侧边栏等）。
- 提供 2-3 种区域排列方案，每种方案均附有基于游戏概念中平台和输入情境的理由。
- 询问："这些方案中有符合您脑海中想象的吗？还是我们来构建自定义排列？"

**子部分 3——组件清单**：
- 对于每个区域，列出其包含的 UI 组件。对于每个组件，注明：
  - 组件类型（按钮、列表、卡片、数值显示、输入框等）
  - 它显示的内容
  - 是否可交互
  - 是否使用模式库中的现有模式（按模式名称引用）
  - 是否引入新模式（标记供后续添加到库中）

**子部分 4——ASCII 线框图**：
- 提议根据区域布局和组件列表生成 ASCII 线框图。
- 使用 `AskUserQuestion`："是否在本规格中包含 ASCII 线框图？"
  - 选项："是，包含"、"不，我会附加单独的文件"
- 若是，先在对话中生成线框图，请求反馈后再写入文件。

---

#### D 部分：状态与变体

引导用户超越只考虑理想路径。

**需要询问的问题**（逐一进行）：
- "玩家第一次看到此屏幕时，还没有任何数据，它会是什么样子？（空状态）"
- "当某些事情出错时——错误、失败的操作、缺少资源——会发生什么？（错误状态）"
- "此屏幕上是否存在加载等待？若是，显示什么？（加载状态）"
- "是否有玩家进度状态会改变此屏幕显示的内容？例如锁定内容、付费内容或教程模式叠加层？"
- "此屏幕在任何支持的平台上行为是否不同？（平台变体）"

将收集到的状态呈现为表格供审批：

| 状态/变体 | 触发器 | 变化内容 |
|-----------|--------|----------|
| 默认 | 正常加载 | — |
| 空 | 无可用数据 | [内容区域描述] |
| [等] | [触发器] | [变化内容] |

---

#### E 部分：交互地图

对于布局规格中确定的每个可交互组件，定义：
- 操作（点击、按下、长按、滚动、拖拽）
- 触发它的平台输入（鼠标点击、手柄 A 键、键盘 Enter）
- 即时反馈（视觉、音频、触觉）
- 结果（导航目标、状态变化、数据写入）

使用第 2h 阶段从 `technical-preferences.md` 加载的输入方式——
不要再次询问用户。预先说明："正在映射以下输入方式的交互：[技术偏好中的输入方式]。
覆盖 [手柄支持] 手柄支持。"

逐个处理组件，而不是一次性要求全部。
对于导航操作（进入另一个屏幕），验证目标是否与现有 UX 规格匹配，
或将其标记为规格依赖项。

---

#### E2 部分：触发的事件

对于交互地图中的每个玩家操作，记录游戏或分析系统应触发的相应事件——
或明确说明"无事件"（若不适用）。

**需要询问的问题**：
- "对于每个操作，游戏是否应触发分析事件、游戏状态变化，还是两者都有？"
- "是否有操作**不应该**触发事件——这是否是故意的选择？"

呈现为与交互地图并列的表格：

| 玩家操作 | 触发的事件 | 负载/数据 |
|----------|------------|-----------|
| [操作] | [EventName] 或无 | [随事件传递的数据] |

标记任何修改持久性游戏状态的操作（存档数据、进度、经济）——
这些需要架构团队的明确关注。

---

#### E3 部分：过渡与动画

指定屏幕进入和退出的方式，以及它如何响应状态变化。

**需要询问的问题**：
- "此屏幕如何出现？（淡入、从右滑入、即时弹出、从按钮缩放）"
- "它如何消失？（淡出、滑回、切换）"
- "屏幕内是否有需要动画的状态过渡？（加载动画、成功状态、错误闪烁）"
- "是否有可能引起晕动症的动画——游戏是否有减少动画的选项？"

最低要求：
- 屏幕进入过渡
- 屏幕退出过渡
- 若屏幕有多种状态，至少一个状态变化动画

---

#### F 部分：数据需求

与第 2 阶段收集的 GDD UI 需求部分进行交叉参考。

对于屏幕显示的每条信息，询问：
- "这些数据来自哪里？哪个系统拥有它？"
- "此屏幕需要回写数据，还是只读？"
- "这些数据中是否有时间敏感或实时的？（血量条、冷却计时器）"

将 UI 需要拥有或管理游戏状态的任何情况标记为架构关注点。
UX 规格定义 UI 需要什么；不规定数据如何交付。这是架构决策。

将数据需求呈现为表格：

| 数据 | 来源系统 | 读/写 | 备注 |
|------|----------|-------|------|
| [项目] | [系统] | 读 | — |
| [项目] | [系统] | 写 | [如有关注点] |

---

#### G 部分：无障碍

若 `design/accessibility-requirements.md` 存在，与其交叉参考。

遍历 ux-designer 代理对此屏幕的标准检查清单：
- 所有交互元素的纯键盘导航路径
- 手柄导航顺序（如适用）
- 文本对比度和最小可读字体大小
- 独立于颜色的信息传达（不能仅靠颜色传递信息）
- 非文本元素的屏幕阅读器注意事项
- 需要减少动画替代方案的动效

使用 `AskUserQuestion` 提出无障碍等级的待解问题：
- "项目的无障碍等级是否已确定？"
  - 选项："是，从需求文档读取"、"尚未——作为待解问题标记"、"暂时跳过无障碍部分"

---

#### H 部分：本地化注意事项

记录影响本屏幕在文字翻译时行为的约束。

**需要询问的问题**：
- "此屏幕上哪些文字元素最长？布局可容纳的最大字符数是多少？"
- "是否有文字长度关键影响布局的元素——例如必须保持在一行的按钮标签？"
- "是否有显示数字、日期或货币且需要特定区域格式化的元素？"

注意：目标是标记出任何 40% 文字扩展（英语翻译成德语或法语时常见）
会破坏布局的元素。将这些标记为本地化工程师的高优先级事项。

---

#### I 部分：验收标准

写至少 5 条具体、可测试的标准，QA 测试员不用阅读任何其他设计文档即可验证。
这些将成为 `/story-done` 的通过/失败条件。

**格式**：使用复选框。每条标准必须由人工测试员可验证：

```
- [ ] 从 [触发器] 开始，屏幕在 [X]ms 内打开
- [ ] [元素] 在 [最小] 和 [最大] 值时正确显示
- [ ] [导航操作] 正确路由到 [目标屏幕]
- [ ] 当 [条件] 时出现错误状态，并显示 [具体消息或图标]
- [ ] 键盘/手柄导航以逻辑顺序到达所有交互元素
- [ ] 满足 [无障碍需求]——例如"所有交互元素均有焦点指示器"
```

**最低要求**：
- 1 条性能标准（加载/打开时间）
- 1 条导航标准（至少验证一个入口或出口路径）
- 1 条错误/空状态标准
- 1 条无障碍标准（根据承诺的等级）
- 1 条针对本屏幕核心目的的具体标准

询问用户确认："这些标准是否涵盖了让此屏幕在您的 QA 流程中真正'完成'所需的内容？"

---

### 部分指导：HUD 设计模式

HUD 设计的顺序与 UX 规格模式不同。从哲学开始；
信息架构完成之前不要涉及布局。

#### A 部分：HUD 哲学

请用户用 1-2 句话描述游戏与屏幕信息的关系。

提供框架示例以帮助思考：
- "几乎无 HUD——氛围需要无遮挡的沉浸（如 Hollow Knight、Firewatch）"
- "最小但存在——只有关键信息可见，其他全为情境式（如 Dark Souls）"
- "信息密集——所有决策相关数据始终可见（如 Diablo IV、StarCraft II）"
- "自适应——HUD 密度响应战斗状态、探索模式、菜单（如 God of War）"

此哲学成为所有后续 HUD 决策的设计约束。
若提议的元素与所述哲学冲突，明确指出该冲突。

---

#### B 部分：信息架构

在任何布局工作之前完成此部分，不要跳过。

**步骤 1——完整信息清单**：
从第 2 阶段收集的 GDD UI 需求部分提取所有信息。
呈现完整列表："这是您的游戏系统表示需要在屏幕上向玩家传达的所有内容。"

**步骤 2——分类**：
对于每个项目，请用户进行分类：

| 类别 | 描述 |
|------|------|
| **必须显示** | 始终可见，玩家需要它做出核心决策 |
| **情境式** | 仅在相关时可见（战斗中、靠近可交互物体时等） |
| **按需** | 玩家必须主动请求（切换、长按按键） |
| **隐藏** | 通过世界/音频传达，从不以文字显示 |

使用 `AskUserQuestion` 每次处理 3-4 个项目，而不是一次全部处理。
这是 HUD 中最重要的设计决策——不要仓促处理。

**冲突检查**：若信息哲学（A 部分）说"几乎无 HUD"
但"必须显示"列表越来越长，明确指出冲突：
> "当前'必须显示'列表有 [N] 个项目。这可能与无 HUD 哲学冲突。
> 选项：缩减'必须显示'列表、修订哲学，或定义在探索时无 HUD、战斗时有 HUD 的混合方法。"

---

#### C 部分：布局区域

仅在信息架构获批后，才设计布局区域。

基础布局考虑：
- 哪些项目是"必须显示"（它们驱动永久区域决策）
- 玩家在游戏中自然注意力所在（动作游戏的屏幕中央、策略游戏的角落）
- 目标平台和宽高比

提供 2-3 种区域排列方案，附上基于 HUD 哲学和 B 部分分类的理由。

---

#### D 部分：HUD 元素

对于布局中的每个元素，指定：
- 元素名称和类别（必须显示/情境式/按需）
- 显示的内容
- 视觉形式（条形、数字、图标、计数器、地图）
- 更新行为（实时、事件驱动、玩家查询）
- 情境触发器（若非始终可见）
- 动画行为（低时是否脉动？淡入？快速出现？）

逐元素处理。若状态显示、资源条或冷却指示器有相关模式存在于模式库中，引用交互模式库。

---

#### E、F、G 部分：动态行为、平台变体、无障碍

这些遵循与 UX 规格等效部分相同的结构。
请参阅 UX 规格部分指导中的 D（状态/变体）、E（交互）和 G（无障碍）。

对于 HUD 特别强调：
- 动态行为：游戏进行中什么原因导致 HUD 密度改变？
- 平台变体：移动/主机是否需要不同的元素尺寸或位置？

---

### 部分指导：交互模式库模式

模式库编写是累加的、目录驱动的，而非线性的。

#### 第 1 阶段：目录现有模式

Glob `design/ux/*.md`（不包括 `interaction-patterns.md`），读取每个规格的
组件清单和交互地图部分，提取每个交互模式。

呈现提取的列表："根据现有 UX 规格，这些模式已在游戏中使用：
- [模式名称]：在 [屏幕]、[屏幕] 中使用
- [等]"

询问："是否有您知道存在但尚未在规格中的模式？现在列出任何额外的模式。"

---

#### 第 2 阶段：正式化每个模式

对于每个模式（现有或新的），记录：

```markdown
### [Pattern Name]

**Category**: Navigation / Input / Feedback / Data Display / Modal / Overlay / [other]
**Used In**: [list of screens]
**Description**: [One paragraph explaining what this pattern is and when to use it]

**Specification**:
- [Component behavior]
- [Input mapping]
- [Visual/audio feedback]
- [Accessibility requirements for this pattern]

**When to Use**: [Conditions where this pattern is appropriate]
**When NOT to Use**: [Conditions where another pattern is more appropriate]

**Reference**: [Screenshot path or ASCII example, if available]
```

批量处理模式。提问："是否需要我根据现有规格中发现的内容起草第一批模式，
还是逐一定义？"

---

#### 第 3 阶段：识别缺口

目录已知模式后，询问：
- "是否有计划中的屏幕或交互需要本库中尚不存在的模式？"
- "现有规格中是否有彼此不一致、应该合并的模式？"

在缺口部分记录缺口供后续跟进。

---

## 5. 交叉参考检查

在将规格标记为可审阅之前，运行以下检查：

**1. GDD 需求覆盖**：引用本屏幕的每条 GDD UI 需求是否在本规格中有对应元素？呈现所有缺口。

**2. 模式库对齐**：本规格使用的所有交互模式是否均按名称引用？
若在本规格会话中发明了新模式，将其标记为需添加到模式库：
> "本规格使用 [模式名称]，但该模式尚不在模式库中。
> 现在添加，还是标记为缺口？"

**3. 导航一致性**：本规格中的入口/出口点是否与相关规格中的导航地图匹配？标记不匹配项。

**4. 无障碍覆盖**：本规格是否满足 `design/accessibility-requirements.md` 中承诺的无障碍等级？
若否，标记待解问题。

**5. 空状态**：每个依赖数据的元素是否都定义了空状态？标记未定义的项目。

呈现检查结果：
> **交叉参考检查：[屏幕名称]**
> - GDD 需求：[覆盖 M 条中的 N 条 / 全部已覆盖]
> - 需添加到库中的新模式：[列表或"无"]
> - 导航不匹配：[列表或"无"]
> - 无障碍缺口：[列表或"无"]
> - 缺少空状态：[列表或"无"]

## 8. 专业代理路由

本技能使用 `ux-designer` 作为主要代理（在前置内容中设置）。对于特定子主题，
可能需要额外的上下文或协调：

| 主题 | 协调对象 |
|------|----------|
| 视觉美学、颜色、布局感 | `art-director`——UX 规格定义区域；美术定义外观 |
| 实施可行性（引擎约束） | `ui-programmer`——最终确定组件清单之前 |
| 游戏数据需求 | `game-designer`——当数据所有权不明确时 |
| UI 中可见的叙事/传说 | `narrative-director`——用于风味文字、物品名称、传说面板 |
| 无障碍等级决策 | 由本会话处理——由 ux-designer 拥有 |

通过 Task 工具委派给其他代理时：
- 提供：屏幕名称、游戏概念摘要、需要专家输入的具体问题
- 代理返回分析到本会话
- 本会话向用户呈现代理的输出
- 用户决策；本会话写入文件
- 代理不直接写入文件——本会话拥有所有文件写入

## 协作协议

本技能在每一步都遵循协作设计原则：

1. 每个部分都遵循**问题 -> 选项 -> 决策 -> 草稿 -> 审批**
2. 在每个决策点使用 **AskUserQuestion**（解释 -> 记录模式）：
   - 第 2 阶段："准备开始，还是需要更多上下文？"
   - 第 3 阶段："我可以创建骨架吗？"
   - 第 4 阶段（每个部分）：设计问题、方法选项、草稿审批
   - 第 5 阶段："运行交叉参考检查？下一步是什么？"
3. 在骨架和每个部分写入之前询问 **"我可以写入 [filepath] 吗？"**
4. **增量写入**：每个部分审批后立即写入文件
5. **会话状态更新**：每次写入部分后更新

**美学尊重**：当布局或视觉选择取决于个人喜好时，呈现选项并询问。
不要因为"标准"而选择某个布局——始终确认。用户是创意总监。

**冲突浮现**：当 GDD 需求与可用屏幕空间冲突时，浮现冲突并呈现解决选项。
永远不要悄悄放弃某个需求。永远不要在不标记的情况下悄悄扩展布局。

**绝不**自动生成完整规格并将其作为既成事实呈现。
**绝不**在未获用户批准的情况下编写某个章节。
**绝不**在未标记冲突的情况下与已批准的 UX 规格相矛盾。
**始终**说明决策来源（GDD 需求、玩家旅程、用户选择）。

结论：**COMPLETE** — UX 规格已逐节编写并获得批准。
