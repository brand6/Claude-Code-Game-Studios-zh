# 平衡检查

平衡检查分析游戏平衡数据以识别可能破坏游戏体验的数值问题——过强的策略、无意义的选择、奖励曲线异常，以及不可持续的经济设计。

---

## 阶段 1：识别平衡域

解析参数以确定范围：
- **`all`**（默认）：运行所有平衡域
- **`combat`**：仅战斗系统（伤害、DPS、血量、防御）
- **`economy`**：仅经济系统（货币、资源流入/流出、成本）
- **`progression`**：仅成长系统（经验值、等级曲线、解锁）
- **`loot`**：仅战利品系统（掉落率、稀有度分布、期望值）
- **`system: <名称>`**：特定系统的深度分析

若未提供参数，检查哪些数据文件实际存在——仅针对有数据支撑的域运行分析。

---

## 阶段 2：读取数据文件

Glob 以下路径中的平衡相关数据：

```
assets/data/balance/**/*.*
assets/data/characters/**/*.*
assets/data/enemies/**/*.*
assets/data/items/**/*.*
assets/data/progression/**/*.*
assets/data/economy/**/*.*
```

读取所有找到的 JSON/YAML/CSV 数据文件。

建立**数值索引**——按系统分类的所有可量化数值。示例：
```
战斗数值：
  玩家 DPS：[15, 18, 22, 45, 12]（根据武器）
  敌人血量：[100, 250, 80, 1500]（根据敌人类型）
  伤害倍率：[1.0, 1.5, 2.0, 0.5]

经济数值：
  物品成本：[10, 25, 50, 200, 5]
  金币掉落：[5, 10, 15]
  商店价格：[30, 75, 150, 500]
```

---

## 阶段 3：读取设计文档

读取以下文件以了解设计意图（平衡评估需要"目标"作为参照基准）：

- `design/gdd/combat.md`——目标战斗感受（若存在）
- `design/gdd/economy.md`——资源/经济意图（若存在）
- `design/gdd/progression.md`——成长设计（若存在）
- `design/gdd/[其他相关 GDD].md`

从每个 GDD 中提取：
- 目标难度曲线表述（"应该有挑战性，但不令人沮丧"等）
- 明确的数值目标（若有记录）
- 战斗/关卡/系统节奏目标（秒级 TTK、每次遭遇预期持续时间等）

---

## 阶段 4：执行分析

对范围内的每个域运行以下检查：

### 4a — 战斗平衡

**DPS 分析：**
计算每种武器/技能的 DPS（每秒伤害）。检查：
- 离群值：DPS 最高的选项是否超过平均值 3 倍以上？（退化策略）
- "最优解"数量：有多少选项的 DPS 互相接近？（有意义的选择）
- DPS 为零的选项（道具栏死荷载）

**TTK 分析（击杀时间）：**
计算典型遭遇中的 TTK：
```
TTK = 目标血量 / 玩家 DPS
```
检查：
- TTK < 1 秒的遭遇（秒杀——感觉不到战斗）
- TTK > 120 秒的遭遇（消耗战——感觉单调）
- 敌人组合中 TTK 的方差（一致性指标）

**防御有效性：**
检查防御/抗性数值是否会对 TTK 造成有意义的影响（10-50% 变化），还是在实际中被忽略。

**CRIT/异常状态倍率：**
检查暴击或异常状态倍率是否在合理范围（1.25x-3.0x 通常为正常）。大于 5x 的倍率通常会破坏常规战斗的意义。

---

### 4b — 经济平衡

**流入/流出分析：**
列举所有金币来源（掉落、任务奖励、出售物品）和消耗途径（购买、升级、制作）：

```
预计每小时流入：[N] 金币
预计每小时流出：[N] 金币
净积累：[+N / -N]
```

检查：
- 严重通货膨胀（流入 > 流出 2 倍以上 → 金币失去意义）
- 严重通货紧缩（流出 > 流入 → 进度阻塞）
- 单一流入来源主导（超过 80%）

**定价层级分析：**
检查商品和服务的价格层级是否遵循逻辑梯度。寻找：
- 价格层级之间的明显缺口（应该在 50-100 金币档的物品，结果是 10 或 500）
- 成本/效益不匹配（价格相同但能力天差地别的物品）

**资源稀缺性：**
对于每种主要资源，检查玩家能否以合理的游戏节奏积累足够的资源来推进进度——还是会遇到阻碍正常游戏的人为稀缺。

---

### 4c — 成长平衡

**等级曲线分析：**
提取每级所需的经验值，计算增长率：

```
Δ经验值[N] = 所需经验值[N+1] / 所需经验值[N]
```

检查：
- 增长率是否一致（1.5x-2.0x 每级通常合理）
- 曲线中的"悬崖"（单一等级所需经验量远超其他）
- 成长终点（最终可用等级的经验量是否与完成内容所能获取的量匹配）

**能力解锁节奏：**
列出解锁里程碑。检查：
- 解锁间距是否均匀（避免"什么都没有等了 10 个等级终于得到一个东西"）
- 建立"感觉良好"的解锁节奏（大里程碑之间有小里程碑支撑）

**能力强度扩展：**
比较每个等级阶段的能力强度（早期、中期、后期）。应该有明显增长，但不能大到使早期内容显得微不足道。

---

### 4d — 战利品平衡

**掉落率审计：**
对所有战利品表提取掉落率。检查：
- 总和是否等于 100%（或接近 100%，若包含"无掉落"）
- 稀有物品的掉率是否与其定义的稀有度等级匹配
- 是否有期望值（单次掉落的平均价值）极高的异常物品

**稀有度分布：**
统计每个稀有度层级的物品数量（普通/稀有/史诗/传说等）。检查：
- 分布是否遵循漏斗形（普通 > 稀有 > 史诗 > 传说）
- 某个层级是否过度拥挤（稀有物品太多，失去稀缺感）

**掉率期望值：**
对关键玩家路径（必须完成的遭遇），估算获得关键物品的平均尝试次数。
- 如果某个必需物品期望 > 50 次尝试才能得到，这是一个严重的进度阻碍点。

## 阶段 6：修复与验证循环

报告展示后，询问：
> "是否希望立即修复这些平衡问题？"

如果是：
- 询问优先处理哪个问题（参照建议表中的优先级行）
- 引导用户更新 `assets/data/` 中的相关数据文件或 `design/balance/` 中的公式
- 每次修复后，提议重新运行相关平衡检查，验证没有引入新的异常值
- 如果修复更改了 GDD 中定义的或 ADR 引用的调整旋钮，提醒用户：
  > "此数值在设计文档中有定义。在提交之前，对受影响的 GDD 运行 `/propagate-design-change [path]` 以查找下游影响。"

如果否：
- 汇总未解决问题，建议将报告保存到 `design/balance/balance-check-[system]-[date].md` 供后续处理

最后以此结束：
> "修复后重新运行 `/balance-check` 以验证。"
