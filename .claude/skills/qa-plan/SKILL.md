# QA 计划

本技能为冲刺、功能或单个故事生成结构化 QA 计划。读取所有范围内的故事文件及其引用的 GDD，按测试类型分类每个故事，产出计划，明确告知开发者哪些需要自动化、哪些需要手动验证、冒烟测试范围是什么、何时需要邀请测试游玩者。

**在冲刺实现开始前运行此技能**，让团队提前了解所需的测试工作。在实现完成后才编写的测试计划是事后总结，而非计划。

**输出：** `production/qa/qa-plan-[sprint-slug]-[date].md`

---

## 阶段1：解析范围

**参数：** `$ARGUMENTS`（空白 = 通过 AskUserQuestion 询问用户）

根据参数确定范围：

- **`sprint`** — 读取 `production/sprints/` 中最新的文件，提取所有引用的故事文件路径。若 `production/sprint-status.yaml` 存在，以其为主要故事列表，以冲刺计划作为故事元数据的补充。
- **`feature: [system-name]`** — Glob `production/epics/*/story-*.md`，过滤文件路径或标题包含系统名称的故事。同时检查该系统目录下的 epic 索引文件（`EPIC.md`）。
- **`story: [path]`** — 验证路径存在并加载该单个文件。
- **无参数** — 使用 `AskUserQuestion`：
  - "此 QA 计划的范围是什么？"
  - 选项："当前冲刺"、"特定功能（输入系统名称）"、"特定故事（输入路径）"、"完整 Epic"

确定范围后，报告："正在为 [范围] 中的 [N] 个故事构建 QA 计划。"

若引用的故事文件路径不存在，将其标注为 MISSING 并继续处理其余故事，不因单个缺失文件而中止整个计划。

---

## 阶段2：加载输入

对每个范围内的故事文件，完整读取并提取：

- **故事标题**和故事 ID（来自文件名或标题）
- **故事类型**字段（若文件头中存在，如 `Type: Logic`）
- **验收标准** — 完整的编号/项目符号列表
- **实现文件** — 列于"待创建/修改文件"或类似章节下
- **引擎备注** — 任何引擎 API 警告或版本特定说明
- **GDD 引用** — 引用的 GDD 路径
- **ADR 引用** — 引用的 ADR
- **估算** — 小时数或故事点（若存在）
- **依赖项** — 该故事依赖的其他故事

读取故事后，加载一次辅助上下文（不对每个故事重复加载）：

- `design/gdd/systems-index.md` — 了解系统优先级以及哪些 GDD 已审批
- 对所有故事中引用的每个唯一 GDD：只读取**验收标准**和**公式**章节，不加载完整 GDD——这两个章节包含可测试需求和需要验证的数学内容
- `docs/architecture/control-manifest.md` — 扫描自动化测试应防范的禁止模式（若文件存在）

若故事中没有引用 GDD，标注为缺口但不阻塞计划。该故事将仅基于验收标准进行分类。

---

## 阶段3：分类每个故事

对每个故事分配故事类型。若故事文件头中已有 `Type:` 字段，使用该值并用以下标准验证；若字段缺失或模糊，根据验收标准推断类型。

| 故事类型 | 分类指标 |
|--------|--------|
| **逻辑** | 验收标准涉及计算、公式、数值阈值、状态转换、AI 决策、数据验证、增益/减益叠加、经济交易或任何可测试的计算逻辑 |
| **集成** | 标准涉及两个或多个系统的交互、信号或事件跨系统边界传播、存档/读档往返、网络同步或持久化 |
| **视觉/手感** | 标准涉及动画行为、VFX、Shader 输出、"响应感"、感知时机、屏幕震动、粒子效果、音频同步或视觉反馈质量 |
| **UI** | 标准涉及菜单、HUD 元素、按钮、界面、对话框、物品栏面板、提示框或任何面向玩家的界面元素 |
| **配置/数据** | 变更仅限于平衡调整数值、数据文件或配置，不涉及新代码逻辑 |

**混合故事**（如同时添加公式和 UI 显示的故事）：根据哪类验收标准的实现风险最高来分配主要类型，并注明次要类型。逻辑+集成和视觉+UI 的混合是最常见的组合。

分类完所有故事后，在进入阶段4前，先在会话中呈现分类汇总表，让用户了解测试资源的分配情况。

---

## 阶段4：生成测试计划

组装完整的 QA 计划文档，使用以下结构：

````markdown

**日期**：[日期]
**生成工具**：/qa-plan
**范围**：[[N] 个故事，覆盖 [N] 个系统]
**引擎**：[来自 .claude/docs/technical-preferences.md 的引擎名称，或"未配置"]
**冲刺文件**：[冲刺计划路径（若适用）]

## 冒烟测试范围

本次冲刺 QA 移交前需验证的关键路径：

1. 游戏启动至主菜单无崩溃
2. 新游戏 / 新会话可以启动
3. [本次冲刺引入或变更的主要机制]
4. [本次冲刺变更中有回归风险的系统]
5. 存档/读档循环完成且数据无丢失（若存档系统已存在）
6. 目标硬件上性能在预算内（无新的帧率峰值）

*冒烟测试由开发者通过 `/smoke-check` 验证。运行该技能时参考此列表。*

| 故事 | 测试游玩目标 | 最少场次 | 目标玩家类型 |
|------|-----------|--------|-----------|
| [故事] | [本次会话需要回答什么问题？] | [N] | [新手玩家 / 有经验玩家] |

**审批要求**：测试游玩记录必须写入 `production/session-logs/playtest-[sprint]-[story-slug].md`，并经 [设计师 / qa-lead] 审查后，故事才可标记为 COMPLETE。

## 完成定义——本次冲刺

满足以下**所有**条件时，故事才算 DONE：

- [ ] 所有验收标准已验证——通过自动化测试结果或书面手动证据（截图、视频或经审批的测试游玩记录）
- [ ] 所有逻辑和集成类故事的测试文件存在于指定路径
- [ ] 所有视觉/手感和 UI 类故事的手动证据文档已存在
- [ ] 冒烟检查通过（QA 移交前运行 `/smoke-check sprint`）
- [ ] 未引入回归
- [ ] 代码已审查（通过 `/code-review` 或书面同行评审）
- [ ] 故事文件已通过 `/story-done` 更新为 `Status: Complete`
````

生成内容时，使用阶段2中提取的实际故事标题、GDD 公式文本和验收标准，不使用占位文本——每个测试条目都应反映这些特定故事的真实需求。

---

## 阶段5：写入输出

在会话中展示完整计划（若计划较长则展示摘要），然后询问：

"可以将此 QA 计划写入 `production/qa/qa-plan-[sprint-slug]-[date].md` 吗？"

按生成内容原样写入，不得截断。

写入后：

"QA 计划已写入 `production/qa/qa-plan-[sprint-slug]-[date].md`。

下一步：
- 冲刺实现开始前将此计划分享给团队
- 所有故事实现完成后，运行 `/smoke-check sprint` 作为 QA 移交的门控
- 对于逻辑/集成类故事，在标记故事完成前在指定路径创建测试文件——`/story-done` 会检查这些文件"

---

## 协作协议

- **未经询问绝不写入计划** — 阶段5需要明确审批。
- **保守分类**：当故事在逻辑和集成之间模糊时，归类为集成——它同时需要单元测试和集成测试。
- **不凭空发明测试用例**，超出验收标准和 GDD 公式所支持的范围。若 GDD 中缺少公式，标注为缺口而非猜测。
- **测试游玩要求为建议性**：用户决定对于边界的视觉/手感故事是否需要测试游玩。标注该情况，不强制要求。
- 在未提供参数时使用 `AskUserQuestion` 进行范围选择。其他所有阶段保持非交互式——呈现发现，然后询问一次以审批写入。
