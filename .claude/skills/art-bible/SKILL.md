## 阶段 2：视觉身份基础（第 1-4 章节）

这四个章节定义核心视觉语言。**所有其他章节都从它们衍生。** 逐章节编写并写入文件，完成一章后再进行下一章。

### 第 1 章：视觉身份声明

**目标**：一句话视觉规则，加上 2-3 条支撑原则，用于解决视觉上的歧义。

若 game-concept.md 中存在视觉锚点，展示并询问：
- "直接基于此锚点构建？"
- "在展开之前先修订它？"
- "重新开始，提供新的方向？"

**Agent 委托（强制）**：通过 Task 生成 `art-director`：
- 提供：游戏概念（核心幻想、一句话定位）、完整支柱集、目标平台、阶段 1 框架定位中的任何参考游戏/美术、视觉锚点（若存在）
- 要求："为这款游戏起草一份视觉身份声明。提供：（1）一句话视觉规则，能够解决任何视觉决策的歧义；（2）2-3 条支撑视觉原则，每条附带一句设计测试（'当 X 有歧义时，此原则说选 Y'）。所有原则必须直接锚定在所述支柱上——每条原则必须服务于一个特定支柱。"

向用户展示美术总监的草稿。使用 `AskUserQuestion`：
- 选项：`[A] 确认锁定` / `[B] 修改一句话规则` / `[C] 修改某条支撑原则` / `[D] 描述我自己的方向`

立即将批准的章节写入文件。

### 第 2 章：氛围与情绪

**目标**：按游戏状态划分的情感目标——具体到光照艺术家可以直接参照工作。

对于每个主要游戏状态（例如探索、战斗、胜利、失败、菜单——根据此游戏的状态进行调整），定义：
- 主要情感/情绪目标
- 光照特征（时段、色温、对比度）
- 氛围描述词（3-5 个形容词）
- 能量等级（急促 / 平稳 / 沉思 / 等）

**Agent 委托**：通过 Task 生成 `art-director`，提供视觉身份声明和支柱集。要求："为这款游戏定义每个主要游戏状态的氛围和情绪目标。要具体——'黑暗且不祥'不够用。说明精确的情感目标、光照特征（暖/冷、高/低对比度、时段方向），以及至少一个承载氛围的视觉元素。每个游戏状态在视觉上必须与其他状态明显区分。"

立即将批准的章节写入文件。

### 第 3 章：形状语言

**目标**：使这款游戏的世界在视觉上保持连贯且与众不同的几何词汇。

涵盖：
- 角色轮廓设计理念（缩略图尺寸下的可读性？每种原型的区分特征？）
- 环境几何（棱角/曲线/有机/几何——哪个主导，为什么？）
- UI 形状语法（UI 是否与世界美学呼应，还是独立的 HUD 语言？）
- 主体形状 vs 衬托形状（什么吸引眼球，什么退为背景？）

**Agent 委托**：通过 Task 生成 `art-director`，提供视觉身份声明和氛围目标。要求："为这款游戏定义形状语言。将每条形状原则与视觉身份声明和特定游戏支柱相连接。解释这些形状选择向玩家传达了什么情感。"

立即将批准的章节写入文件。

### 第 4 章：色彩系统

**目标**：一个完整的、可付诸生产的调色板系统，同时满足美学需求和传达功能。

涵盖：
- 主调色板（5-7 种颜色，附带各自的功能角色——不只是十六进制代码，而是每种颜色在这个世界中的含义）
- 语义色彩用法（红色传达什么？金色？蓝色？白色？建立色彩词汇）
- 按生物群系或区域划分的色温规则（若游戏有不同区域）
- UI 调色板（可能与世界调色板不同——明确定义差异）
- 色盲安全性：哪些语义色彩需要形状/图标/声音作为备用

**Agent 委托**：通过 Task 生成 `art-director`，提供视觉身份声明和氛围目标。要求："为这款游戏设计色彩系统。每项语义色彩分配都必须加以解释——为什么这种颜色在这个世界中意味着危险/安全/奖励？识别哪些色彩搭配可能对色盲玩家不友好，并说明需要什么备用提示。"

立即将批准的章节写入文件。

## 阶段 3：制作指南（第 5-8 章节）

这些章节将视觉身份转化为具体的制作规则。应足够具体，让外包团队无需额外说明即可遵循。

### 第 5 章：角色设计方向

**Agent 委托**：通过 Task 生成 `art-director`，提供第 1-4 章节内容。要求："为这款游戏定义角色设计方向。涵盖：玩家角色的视觉原型（若有）、每种角色类型的区分特征规则（玩家如何一眼区分敌人/NPC/盟友？）、表情/动作风格目标（僵硬/富有表现力/写实/夸张），以及 LOD 设计理念（游戏摄像机距离下保留多少细节？）。"

将批准的章节写入文件。

### 第 6 章：环境设计语言

**Agent 委托**：通过 Task 生成 `art-director`，提供第 1-4 章节内容。要求："为这款游戏定义环境设计语言。涵盖：建筑风格及其与世界文化/历史的关系、材质理念（手绘 vs PBR vs 程式化——为什么这款游戏选这种？）、道具密度规则（疏密——每种区域类型的选择依据是什么？），以及环境叙事指南（哪些视觉细节应在无文字的情况下讲述故事？）。"

将批准的章节写入文件。

### 第 7 章：UI/HUD 视觉方向

**Agent 委托**：并行生成：
- **`art-director`**：UI 视觉风格——叙事内嵌式 vs 屏幕空间 HUD、排版方向（字体个性、字重、尺寸层级）、图标风格（扁平/描边/插画/写实）、UI 元素的动画感
- **`ux-designer`**：UX 对齐检查——视觉方向是否支持这款游戏所需的交互模式？标记美术方向与可读性/可访问性需求之间的任何冲突。

收集两方意见。若存在冲突（例如美术总监想要精美的叙事内嵌式 UI，但 UX 设计师指出会降低战斗可读性），明确提出冲突并呈现双方立场。不要静默解决——使用 `AskUserQuestion` 让用户决策。

将批准的章节写入文件。

### 第 8 章：资产规范

**Agent 委托**：并行生成：
- **`art-director`**：文件格式偏好、命名规范方向、材质分辨率层级、LOD 级别预期、导出设置理念
- **`technical-artist`**：引擎特定硬性约束——每类资产的面数预算、材质内存限制、材质插槽数量、导入器约束、`.claude/docs/technical-preferences.md` 中性能预算的任何约束

若任何美术偏好与技术约束冲突（例如美术总监想要 4K 材质，但移动平台性能预算要求 2K），明确解决冲突——注明理想标准和受约束标准，并解释权衡。资产规范的歧义是制作成本的来源。

将批准的章节写入文件。

## 阶段 4：参考方向（第 9 章节）

**目标**：一组精心挑选的参考资料，具体说明从每个来源取什么、不取什么。

**Agent 委托**：通过 Task 生成 `art-director`，提供已完成的第 1-8 章节内容。要求："为这款游戏整理一份参考方向。提供 3-5 个参考来源（游戏、电影、美术风格或特定艺术家）。对每个来源：说明名称，指定要借鉴的具体视觉元素（不是'整体美学'——而是特定技法、色彩选择或构图规则），并指定明确要避免或偏离的内容（以防'试图模仿 X'的观感）。参考资料应是互补的——任意两个参考不应指向完全相同的方向。"

将批准的章节写入文件。

## 协作协议

每个章节遵循以下流程：**提问 → 选项 → 决定 → 草稿（来自 art-director 代理）→ 批准 → 写入文件**

- 在召唤相关代理之前，绝不起草任何章节内容
- 每个章节获批后立即写入文件，不要批量处理
- 将所有代理间的分歧呈现给用户——不要在 art-director 和 technical-artist 之间悄悄解决冲突
- 美术圣经是约束文档：它以换取视觉一致性为代价，限制未来的决策。每个章节都应感觉像是在有效缩小解决方案空间。

- 运行 `/map-systems`，在编写 GDD 之前将概念分解为游戏系统
