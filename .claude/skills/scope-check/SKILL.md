# 范围审查

本技能是一项**只读**快速范围健康评估——不创建、不更新任何文件，只生成报告。

allowed-tools: Read, Glob, Grep, Bash

## 阶段 1：找到原始计划

读取以下文件（优先级从高到低）：
1. `production/sprints/sprint-[N].md`（迭代计划文件）
2. `production/milestones/` 中当前里程碑的功能范围
3. `design/gdd/systems-index.md` 中相关功能的 MVP 层级定义

记录**原始承诺**：
- 原始用户故事列表（及其点数）
- 原始目标陈述
- 里程碑阶段（Alpha / Beta / Gold）

若找不到原始计划，告知用户："无法找到原始范围基线。请先运行 `/sprint-plan` 为本迭代建立记录。"

---

## 阶段 2：读取当前状态

读取以下文件：
1. Glob `production/stories/**/*.md`，筛选出属于本迭代/功能的用户故事
2. 检查是否存在 `production/session-state/` 下与本迭代相关的文件
3. Grep 迭代标签或功能名称，找出在初始迭代计划编写后新增的故事

记录**当前实际范围**：
- 所有用户故事（包括原计划之外新增的）
- 每个故事的状态（Complete / In Progress / Blocked / Not Started）
- 点数分布（若有点数信息）

---

## 阶段 3：比对范围

将当前范围与原始范围进行对比，识别：

**新增内容：**
- 完全不在原始计划中的新增用户故事
- 用户故事描述中悄然增大的任务（在原始故事文件中写明但超出原有范围）
- 新增的系统或文件，这些内容不在原始计划范围内

**被删除 / 降级的内容：**
- 计划中但已移出本迭代的故事
- 从 MVP 降级为可选的功能

**范围变更量化：**
- 原始故事数量：[N]
- 当前故事数量：[N]（含新增：[+N]，移除：[-N]）
- 净变更率：（当前 - 原始）/ 原始 × 100%
- 若有点数信息：原始承诺点数 vs 当前承诺点数

---

## 阶段 4：裁定

根据以下阈值给出裁定：

| 净变更率 | 裁定 | 含义 |
|---------|------|------|
| ≤ 10% | **PASS** | 正常的范围调整，在合理范围内 |
| 10–25% | **CONCERNS** | 需要关注的范围蔓延——需要制作人审查 |
| > 25% | **FAIL** | 范围蔓延严重——强烈建议削减范围 |
| > 50% 的故事为新增 | **FAIL** | 范围已被根本性改变，需要重新规划 |

---

## 阶段 5：展示报告

```
## 范围审查结果：[迭代/功能名称]

**裁定**：[PASS | CONCERNS | FAIL]
**净范围变更**：[+N / -N]（[百分比]%）

### 新增内容（非计划）
- [故事名称] — [状态] — [关联系统]
- [故事名称] — [状态] — [关联系统]
（如无则写"未检测到范围外新增内容"）

### 被移除 / 降级的内容
- [故事名称] — 已移至下一迭代 / 已降级为可选
（如无则写"无"）

### 范围汇总
| | 故事数 | 点数（如有）|
|--|---------|------------|
| 原始承诺 | [N] | [N] |
| 当前实际 | [N] | [N] |
| 净变更 | [+/-N] | [+/-N] |

### 建议
[若 PASS：正常推进。若 CONCERNS 或 FAIL：列出具体建议——比如将哪些故事移出本迭代、召集制作人审查，或重新规划里程碑。]
```

---

## 后续步骤

- **若 PASS**：范围健康，无需立即行动。
- **若 CONCERNS**：建议与制作人讨论是否削减范围，再继续实施。
- **若 FAIL**：运行 `/sprint-plan` 重新规划范围，或运行 `/estimate` 重新估算工期基线，再联系制作人讨论里程碑调整方案。

完成报告后，始终以以下语句结尾：
> "在削减范围后，再次运行 `/scope-check [name]` 以确认裁定结果改善。"
