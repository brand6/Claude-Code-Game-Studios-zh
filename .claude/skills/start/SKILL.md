# 引导式入门

本技能只写入一个文件：`production/review-mode.txt`（在阶段 3b 中设置评审模式配置）。

本技能是新用户的入口点。它**不假设**你已有游戏创意、引擎偏好或任何先前经验。它先提问，再引导你进入正确的工作流。

---

## 阶段 1：检测项目状态

在提问之前，静默收集上下文，以便为你提供针对性的指引。**不要**主动展示这些结果——它们用于辅助你的推荐，而非作为开场内容。

检查以下内容：
- **引擎是否已配置？** 读取 `.claude/docs/technical-preferences.md`。若 Engine 字段包含 `[TO BE CONFIGURED]`，则引擎未配置。
- **游戏概念是否存在？** 检查 `design/gdd/game-concept.md` 是否存在。
- **源代码是否存在？** 使用 Glob 查找 `src/` 中的源文件（`*.gd`、`*.cs`、`*.cpp`、`*.h`、`*.rs`、`*.py`、`*.js`、`*.ts`）。
- **是否有原型？** 检查 `prototypes/` 是否有子目录。
- **是否有设计文档？** 统计 `design/gdd/` 中的 markdown 文件数量。
- **是否有生产工件？** 检查 `production/sprints/` 或 `production/milestones/` 是否有文件。

将这些发现在内部保存，用于验证用户的自我评估并定制推荐内容。

---

## 阶段 2：询问用户当前所处阶段

这是用户第一眼看到的内容。使用 `AskUserQuestion` 提供以下选项，让用户可以直接点击而无需手动输入：

- **提示**："欢迎使用 Claude Code Game Studios！在我提出任何建议之前，我想了解一下你目前的起点。你的游戏创意目前处于哪个阶段？"
- **选项**：
  - `A) 还没有创意` — 我完全没有游戏概念。我想探索一下，想想要做什么。
  - `B) 有个模糊的想法` — 我脑海中有个粗略的主题、感觉或类型（比如"太空题材"或"田园休闲游戏"），但还没有具体内容。
  - `C) 有明确的概念` — 我知道核心创意——类型、基本机制，或许还有一句话的卖点——但还没有正式整理成文档。
  - `D) 已有现成工作` — 我已经有了设计文档、原型、代码或大量规划内容。我想整理或继续这些工作。

等待用户选择。在用户回应之前不要继续。

---

## 阶段 3：根据回答进行路由

#### 若选择 A：还没有创意

用户在任何事情之前都需要创意探索。

1. 表示从零开始完全没问题
2. 简要说明 `/brainstorm` 的作用（使用专业框架进行引导式创意探索——MDA、玩家心理学、动词优先设计）。提及它有两种模式：`/brainstorm open` 适合完全开放式探索，或 `/brainstorm [提示词]` 适合已有模糊主题（如"太空"、"休闲"、"恐怖"）的情况。
3. 推荐以 `/brainstorm open` 作为下一步，但如果他们想到了什么，也欢迎加上提示词
4. 展示推荐路径：
   **概念阶段：**
   - `/brainstorm open` — 探索游戏概念
   - `/setup-engine` — 配置引擎（brainstorm 会推荐一个）
   - `/art-bible` — 定义视觉风格（使用 brainstorm 产生的视觉锚点）
   - `/map-systems` — 将概念分解为各个系统
   - `/design-system` — 为每个 MVP 系统编写 GDD
   - `/review-all-gdds` — 跨 GDD 一致性检查
   - `/gate-check` — 在进入架构工作前验证准备状态
   **架构阶段：**
   - `/create-architecture` — 生成主架构蓝图和必要 ADR 清单
   - `/architecture-decision (×N)` — 按必要 ADR 清单记录关键技术决策
   - `/create-control-manifest` — 将决策整理为可执行规则清单
   - `/architecture-review` — 验证架构覆盖情况
   **预制作阶段：**
   - `/ux-design` — 为关键界面编写 UX 规格（主菜单、HUD、核心交互）
   - `/prototype` — 构建一次性原型以验证核心机制
   - `/playtest-report (×1+)` — 记录每次垂直切片的测试报告
   - `/create-epics` — 将系统映射为功能模块
   - `/create-stories` — 将功能模块拆解为可实现的用户故事
   - `/sprint-plan` — 规划第一次迭代
   **生产阶段：** → 使用 `/dev-story` 开始实现用户故事

#### 若选择 B：有个模糊的想法

1. 请用户分享他们的模糊想法——哪怕几个词也够了
2. 认可这个想法是一个很好的出发点（不要评判或转移话题）
3. 推荐运行 `/brainstorm [他们的提示词]` 来进一步发展
4. 展示推荐路径：
   **概念阶段：**
   - `/brainstorm [提示词]` — 将想法发展为完整概念
   - `/setup-engine` — 配置引擎
   - `/art-bible` — 定义视觉风格（使用 brainstorm 产生的视觉锚点）
   - `/map-systems` — 将概念分解为各个系统
   - `/design-system` — 为每个 MVP 系统编写 GDD
   - `/review-all-gdds` — 跨 GDD 一致性检查
   - `/gate-check` — 在进入架构工作前验证准备状态
   **架构阶段：**
   - `/create-architecture` — 生成主架构蓝图和必要 ADR 清单
   - `/architecture-decision (×N)` — 按必要 ADR 清单记录关键技术决策
   - `/create-control-manifest` — 将决策整理为可执行规则清单
   - `/architecture-review` — 验证架构覆盖情况
   **预制作阶段：**

   - `/create-epics` — 将系统映射为功能模块
   - `/create-stories` — 将功能模块拆解为可实现的用户故事
   - `/sprint-plan` — 规划第一次迭代

#### 若选择 C：有明确的概念

1. 请用户用一句话描述他们的概念——类型和核心机制。使用纯文本，不要 AskUserQuestion（这是开放式回答）。
2. 认可概念后，使用 `AskUserQuestion` 提供两条路径：
   - **提示**："你希望如何继续？"
   - **选项**：
     - `先正式化` — 运行 `/brainstorm [概念]` 将其整理成规范的游戏概念文档
     - `直接开始` — 现在前往 `/setup-engine`，之后再手动编写 GDD
3. 展示推荐路径：
   **概念阶段：**
   - `/brainstorm` 或 `/setup-engine` — （由用户在第 2 步选择）
   - `/art-bible` — 定义视觉风格（在 brainstorm 之后运行，或在概念文档存在之后）
   - `/design-review` — 验证概念文档
   - `/map-systems` — 将概念分解为各个系统
   - `/design-system` — 为每个 MVP 系统编写 GDD
   - `/review-all-gdds` — 跨 GDD 一致性检查
   - `/gate-check` — 在进入架构工作前验证准备状态

   - `/architecture-review` — 验证架构覆盖情况
   **预制作阶段：**
   - `/ux-design` — 为关键界面编写 UX 规格（主菜单、HUD、核心交互）
   - `/prototype` — 构建一次性原型以验证核心机制
   - `/playtest-report (×1+)` — 记录每次垂直切片的测试报告
   - `/create-epics` — 将系统映射为功能模块
   - `/create-stories` — 将功能模块拆解为可实现的用户故事
   - `/sprint-plan` — 规划第一次迭代

#### 若选择 D：已有现成工作

1. 分享阶段 1 中找到的内容：
   - "我可以看到你有 [X 个源文件 / Y 个设计文档 / Z 个原型]..."
   - "你的引擎 [已配置为 X / 尚未配置]..."

2. **子情况 D1 — 早期阶段**（引擎未配置，或仅存在游戏概念）：
   - 如果引擎未配置，推荐先运行 `/setup-engine`
   - 然后运行 `/project-stage-detect` 进行缺口盘点

   **子情况 D2 — 已有 GDD、ADR 或用户故事：**
   - 说明："有文件并不等于模板的技能可以使用它们。GDD 可能缺少必要的章节。`/adopt` 专门检查这一点。"
   - 推荐：
     1. `/project-stage-detect` — 了解当前阶段及完全缺失的内容
     2. `/adopt` — 审计现有工件是否符合正确的内部格式

3. 展示 D2 的推荐路径：
   - `/project-stage-detect` — 阶段检测 + 文件存在性缺口
   - `/adopt` — 格式合规审计 + 迁移计划
   - `/setup-engine` — 如果引擎未配置
   - `/design-system retrofit [路径]` — 填补缺失的 GDD 章节
   - `/architecture-decision retrofit [路径]` — 补充缺失的 ADR 章节
   - `/architecture-review` — 引导建立 TR 需求注册表
   - `/gate-check` — 验证进入下一阶段的准备状态

## 阶段 3b：设置评审模式

检查 `production/review-mode.txt` 是否已存在。

**如果已存在**：读取并显示当前模式——"评审模式当前设置为 `[当前值]`。"——然后继续进入阶段 4。不再重复询问。

**如果不存在**：使用 `AskUserQuestion`：

- **提示**："一个配置选项：在工作流中，你希望进行多少设计评审？"
- **选项**：
  - `完整（Full）` — 专业总监在每个关键工作流步骤进行评审。适合团队协作、学习工作流，或希望对每个决策得到全面反馈时。
  - `精简（Lean，推荐）` — 总监仅在阶段门禁切换时介入（`/gate-check`）。跳过每个技能内部的单独评审。适合独立开发者和小团队的平衡方案。
  - `单人（Solo）` — 完全不进行总监评审。最高速度。适合 Game Jam、原型开发，或感觉评审是额外负担时。

用户选择后，**立即**将结果写入 `production/review-mode.txt`——无需单独询问"是否可以写入"，写入是选择的直接结果：
- `完整（Full）` → 写入 `full`
- `精简（Lean，推荐）` → 写入 `lean`
- `单人（Solo）` → 写入 `solo`

如果 `production/` 目录不存在，则创建它。

---

## 阶段 4：继续前确认

展示推荐路径后，使用 `AskUserQuestion` 询问用户希望先走哪一步。**绝不自动运行下一个技能**。

- **提示**："你想从 [推荐的第一步] 开始吗？"
- **选项**：
  - `是，从 [推荐的第一步] 开始`
  - `我想先做其他事情`

---

## 阶段 5：交接

当用户确认下一步时，用一行简短的话回应："输入 `[技能命令]` 开始。"不加任何其他内容。不要重新解释技能，也不要加鼓励语。`/start` 技能的使命到此完成。

判定：**COMPLETE** — 用户已完成引导，交接至下一步。

---

## 边缘情况

- **用户选择 D 但项目为空**：温和重新引导——"看起来项目是一个尚无工件的全新模板。A 或 B 选项是否更适合你？"
- **用户选择 A 但项目中已有代码**：提及所发现的内容——"我注意到 `src/` 中已有代码。你是否想选择 D（已有现成工作）？"
- **用户再次返回（引擎已配置、概念已存在）**：直接跳过引导——"看起来你已经完成了初始设置！你的引擎是 [X]，游戏概念位于 `design/gdd/game-concept.md`。评审模式：`[从 production/review-mode.txt 读取，若缺失则显示 'lean（默认）']`。想从上次停止的地方继续吗？试试 `/sprint-plan`，或者直接告诉我你想做什么。"
- **用户情况不符合任何选项**：让他们用自己的语言描述情况，灵活应对。

---

## 协作协议

1. **先询问** — 不要预设用户的状态或意图
2. **提供选项** — 给出清晰路径，而非强制要求
3. **用户决定** — 由他们来选择方向
4. **不自动执行** — 推荐下一个技能，不要在未询问的情况下直接运行
5. **灵活应对** — 如果用户的情况不符合模板，倾听并调整
