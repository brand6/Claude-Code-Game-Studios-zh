# 总监门禁 — 共享评审模式

本文档为所有总监及负责人的评审定义了标准门禁 prompt，涵盖每个工作流阶段。
技能通过引用本文档中的门禁 ID 来调用，而非将完整 prompt 内联嵌入——这可消除
prompt 更新时产生的版本漂移。

**适用范围**：全部 7 个生产阶段（概念 → 发布），全部 3 位 Tier 1 总监，
以及所有关键的 Tier 2 负责人。任何技能、团队编排器或工作流均可调用这些门禁。

---

## 如何使用本文档

在任何技能中，将内联的总监 prompt 替换为引用方式：

```
通过 Task 生成 `creative-director`，使用来自
`.claude/docs/director-gates.md` 的门禁 **CD-PILLARS**。
```

传递该门禁 **"需要传入的上下文"** 字段中列出的内容，然后根据以下 **裁定处理** 规则处理结果。

---

## 评审模式

评审强度控制总监门禁是否运行。可全局设置（跨会话持久生效）或在单次技能运行时覆盖。

**全局配置**：`production/review-mode.txt` — 仅一个词：`full`、`lean` 或 `solo`。
在 `/start` 期间设置一次。随时可直接编辑该文件进行修改。

**单次覆盖**：任何使用门禁的技能均接受 `--review [full|lean|solo]` 参数，
仅对本次运行覆盖全局配置。

示例：
```
/brainstorm space horror           → 使用全局模式
/brainstorm space horror --review full   → 本次强制 full 模式
/architecture-decision --review solo     → 本次跳过所有门禁
```

| 模式 | 运行内容 | 适用场景 |
|------|----------|----------|
| `full` | 所有门禁激活——每个工作流步骤均经评审 | 团队协作、新手学习，或需要每步都获得详细总监反馈时 |
| `lean` | 仅 PHASE-GATE（`/gate-check`）——跳过每技能门禁 | **默认** — 独立开发者和小型团队；仅在里程碑节点进行总监评审 |
| `solo` | 任何地方均不设总监门禁 | 开发 Jam、原型验证、追求最高速度 |

**检查模式——在每次生成门禁之前执行：**

```
在生成门禁 [GATE-ID] 之前：
1. 如果技能调用时附带了 --review [mode]，则使用该模式
2. 否则读取 production/review-mode.txt
3. 否则默认为 full

按解析后的模式执行：
- solo → 跳过所有门禁。备注："[GATE-ID] 已跳过 — Solo 模式"
- lean → 仅 PHASE-GATE 执行（CD-PHASE-GATE、TD-PHASE-GATE、PR-PHASE-GATE）；
         其他跳过。备注："[GATE-ID] 已跳过 — Lean 模式"
- full → 正常生成
```

---

## 调用模式（可复制到任意技能中）

**强制要求：在每次生成门禁之前解析评审模式。** 不得在未检查前就生成门禁。解析后的模式在每次技能运行时确定一次：
1. 如果技能调用时附带了 `--review [mode]`，使用该模式
2. 否则读取 `production/review-mode.txt`
3. 否则默认为 `lean`

按解析后的模式执行：
- `solo` → **跳过所有门禁**。在输出中备注：`[GATE-ID] 已跳过 — Solo 模式`
- `lean` → **仅 PHASE-GATE 执行**（CD-PHASE-GATE、TD-PHASE-GATE、PR-PHASE-GATE、AD-PHASE-GATE）。备注：`[GATE-ID] 已跳过 — Lean 模式`
- `full` → 正常生成

```
# 先执行模式检查，然后：
通过 Task 生成 `[agent-name]`：
- 门禁：[GATE-ID]（参见 .claude/docs/director-gates.md）
- 上下文：[该门禁列出的字段]
- 等待裁定后再继续执行。
```

并行生成（多位总监在同一检查点）：

```
# 先对每个门禁执行模式检查，然后生成所有通过检查的门禁：
通过 Task 同时生成全部 [N] 个 Agent——在等待任何结果之前先发出所有 Task 调用。
收集所有裁定后再继续执行。
```

---

## 标准裁定格式

所有门禁返回三种裁定之一。技能必须处理全部三种情况：

| 裁定 | 含义 | 默认处理 |
|------|------|----------|
| **APPROVE / READY** | 无问题，继续执行。 | 继续工作流 |
| **CONCERNS [列表]** | 存在问题但不阻塞。 | 通过 `AskUserQuestion` 告知用户，选项：`修改标记项` / `接受并继续` / `进一步讨论` |
| **REJECT / NOT READY [阻塞项]** | 存在阻塞性问题，不得继续。 | 将阻塞项告知用户。在问题解决前不得写入文件或推进阶段。 |

**升级规则**：当多位总监并行生成时，采用最严格的裁定——一个 NOT READY 将覆盖所有 READY 裁定。

---

## 记录门禁结果

门禁处理完成后，将裁定记录在相关文档的状态头部：

```markdown
> **[总监] 评审（[GATE-ID]）**：APPROVED [日期] / CONCERNS（已接受）[日期] / REVISED [日期]
```

阶段门禁的结果，记录在 `docs/architecture/architecture.md` 或
`production/session-state/active.md`（视情况而定）。

## Tier 1 — 创意总监门禁

Agent：`creative-director` | 模型层级：Opus | 领域：愿景、设计柱、玩家体验

---

### CD-PILLARS — 设计柱压力测试

**触发条件**：游戏设计柱和反设计柱定义完成后（头脑风暴第 4 阶段，或设计柱修订时）

**需要传入的上下文**：
- 完整的设计柱集合，包含名称、定义和设计测试
- 反设计柱列表
- 核心幻想陈述
- 独特卖点（"像 X，并且还有 Y"）

**Prompt**：
> "评审这些游戏设计柱。它们是否可证伪——真实的设计决策是否有可能不符合这些柱？
> 它们之间是否产生有意义的张力？它们是否使本游戏有别于最相近的竞品？在实践中
> 它们是否能帮助解决设计分歧，还是过于模糊而无从参考？针对每根设计柱给出具体
> 反馈，并给出总体裁定：APPROVE（强劲）、CONCERNS [列表]（需要打磨），或
> REJECT（薄弱——设计柱不具分量）。"

**裁定**：APPROVE / CONCERNS / REJECT

---

### CD-GDD-ALIGN — GDD 设计柱对齐检查

**触发条件**：系统 GDD 编写完成后（design-system、quick-design，或任何生成 GDD 的工作流）

**需要传入的上下文**：
- GDD 文件路径
- 游戏设计柱（来自 `design/gdd/game-concept.md` 或 `design/gdd/game-pillars.md`）
- 本游戏的 MDA 美学目标
- 该系统声明的玩家幻想章节

**Prompt**：
> "评审这份系统 GDD 的设计柱对齐程度。每个章节是否都服务于已声明的设计柱？
> 是否存在与设计柱相矛盾或削弱设计柱的机制或规则？玩家幻想章节是否与游戏
> 的核心幻想相符？返回 APPROVE、CONCERNS [存在问题的具体章节]，或 REJECT
> [在该系统可实现前必须重新设计的设计柱违反点]。"

**裁定**：APPROVE / CONCERNS / REJECT

---

### CD-SYSTEMS — 系统分解愿景检查

**触发条件**：`/map-systems` 完成系统索引编写后——在 GDD 编写开始前，验证完整的系统集合

**需要传入的上下文**：
- 系统索引路径（`design/gdd/systems-index.md`）
- 游戏设计柱和核心幻想（来自 `design/gdd/game-concept.md`）
- 优先级层级分配（MVP / 垂直切片 / Alpha / 完整愿景）
- 依赖图中识别出的高风险或瓶颈系统

**Prompt**：
> "对照游戏设计柱评审这份系统分解。MVP 层级的系统集合是否能共同实现核心幻想？
> 是否存在机制不服务于任何已声明设计柱的系统——这可能意味着范围蔓延？是否有
> 设计柱关键的玩家体验没有任何系统来承载？核心循环是否缺少某些必要的系统？
> 返回 APPROVE（系统服务于愿景）、CONCERNS [具体差距或与设计柱的错位及其影响]，
> 或 REJECT [根本性缺口——分解遗漏了关键设计意图，必须在 GDD 编写开始前修订]。"

**裁定**：APPROVE / CONCERNS / REJECT

---

### CD-NARRATIVE — 叙事一致性检查

**触发条件**：叙事 GDD、世界观文档、对话规范或世界构建文档编写完成后
（team-narrative、故事系统的 design-system、writer 交付物）

**需要传入的上下文**：
- 文档文件路径（一个或多个）
- 游戏设计柱
- 叙事方向简报或基调指南（如存在于 `design/narrative/`）
- 新文档引用的任何现有世界观内容

**Prompt**：
> "评审这份叙事内容与游戏设计柱及已有世界规则的一致性。基调是否与游戏已有的
> 风格匹配？是否与现有世界观或世界构建存在矛盾？内容是否服务于玩家体验设计柱？
> 返回 APPROVE、CONCERNS [具体矛盾]，或 REJECT [破坏世界观连贯性的矛盾点]。"

**裁定**：APPROVE / CONCERNS / REJECT

---

### CD-PLAYTEST — 玩家体验验证

**触发条件**：游戏测试报告生成后（`/playtest-report`），或任何产出玩家反馈的会话后

**需要传入的上下文**：
- 游戏测试报告文件路径
- 游戏设计柱和核心幻想陈述
- 本次测试的具体假设

**Prompt**：
> "对照游戏设计柱和核心幻想评审这份游戏测试报告。玩家体验是否与预期幻想一致？
> 是否存在代表设计柱偏移的系统性问题——即机制单独看还好，但整体上削弱了预期
> 体验？返回 APPROVE（核心幻想已准确传达）、CONCERNS [预期与实际体验的差距]，
> 或 REJECT [核心幻想未能呈现——在进行更多游戏测试前需要重新设计]。"

**裁定**：APPROVE / CONCERNS / REJECT

---

### CD-PHASE-GATE — 阶段过渡的创意就绪检查

**触发条件**：在 `/gate-check` 时始终触发——与 TD-PHASE-GATE 和 PR-PHASE-GATE 并行生成

**需要传入的上下文**：
- 目标阶段名称
- 所有已有产出物列表（文件路径）
- 游戏设计柱和核心幻想

**Prompt**：
> "从创意方向角度，评审当前项目状态是否满足进入 [目标阶段] 的就绪标准。
> 游戏设计柱是否在所有设计产出物中得到忠实体现？当前状态是否保留了核心幻想？
> 各 GDD 或架构中是否存在任何设计决策损害了预期玩家体验？返回 READY、
> CONCERNS [列表]，或 NOT READY [阻塞项]。"

**裁定**：READY / CONCERNS / NOT READY

## Tier 1 — 技术总监门禁

Agent：`technical-director` | 模型层级：Opus | 领域：架构、引擎风险、性能

---

### TD-SYSTEM-BOUNDARY — 系统边界架构评审

**触发条件**：`/map-systems` 第 3 阶段依赖图确认后、GDD 编写开始前——在团队
投入编写 GDD 之前验证系统结构在架构上是否合理

**需要传入的上下文**：
- 系统索引路径（如索引尚未编写，则传入依赖图摘要）
- 层级分配（基础层 / 核心层 / 功能层 / 表现层 / 优化层）
- 完整依赖图（每个系统依赖的内容）
- 标记出的瓶颈系统（有大量依赖方）
- 发现的循环依赖及其建议解决方案

**Prompt**：
> "在 GDD 编写开始前，从架构角度评审这份系统分解。系统边界是否清晰——每个系统
> 是否拥有独立的职责，重叠最小化？是否存在上帝对象风险（系统承担了过多职责）？
> 依赖顺序是否会造成实现排期问题？建议的边界中是否存在隐式共享状态问题，在实现
> 时会导致紧耦合？基础层系统是否实际上依赖于功能层系统（依赖倒置）？返回 APPROVE
> （边界在架构上合理——继续编写 GDD）、CONCERNS [需要在 GDD 中解决的具体边界
> 问题]，或 REJECT [根本性边界问题——系统结构将引发架构问题，必须在任何 GDD
> 编写前进行重构]。"

**裁定**：APPROVE / CONCERNS / REJECT

---

### TD-FEASIBILITY — 技术可行性评估

**触发条件**：在范围/可行性评估期间识别出最大技术风险后（头脑风暴第 6 阶段、
quick-design，或任何存在技术未知项的早期概念）

**需要传入的上下文**：
- 概念的核心循环描述
- 目标平台
- 引擎选择（或"未确定"）
- 识别出的技术风险列表

**Prompt**：
> "评审这款 [类型] 游戏使用 [引擎或"未确定引擎"] 在 [平台] 上的技术风险。
> 标记出：可能使概念本身无法成立的 HIGH 风险项、特定于引擎且应影响引擎选型的
> 风险项，以及独立开发者普遍低估的风险项。返回 VIABLE（风险可控）、CONCERNS
> [列表及缓解建议]，或 HIGH RISK [需要修改概念或范围的阻塞项]。"

**裁定**：VIABLE / CONCERNS / HIGH RISK

---

### TD-ARCHITECTURE — 架构确认

**触发条件**：主架构文档草稿完成后（`/create-architecture` 第 7 阶段），以及
任何重大架构修订后

**需要传入的上下文**：
- 架构文档路径（`docs/architecture/architecture.md`）
- 技术需求基线（TR-ID 及数量）
- ADR 列表及状态
- 引擎知识盲区清单

**Prompt**：
> "评审这份主架构文档的技术健壮性。检查：（1）基线中的每项技术需求是否都有
> 对应的架构决策？（2）所有高风险引擎领域是否都有明确应对或标记为开放问题？
> （3）API 边界是否清晰、最小化且可实现？（4）基础层 ADR 的空缺在实现开始前
> 是否已解决？返回 APPROVE、CONCERNS [列表]，或 REJECT [编码开始前必须解决的
> 阻塞项]。"

**裁定**：APPROVE / CONCERNS / REJECT

---

### TD-ADR — 架构决策评审

**触发条件**：单条 ADR 编写完成后（`/architecture-decision`），在标记为"已接受"之前

**需要传入的上下文**：
- ADR 文件路径
- 引擎版本及该领域的知识盲区风险等级
- 相关 ADR（如有）

**Prompt**：
> "评审这份架构决策记录。问题陈述和决策理由是否清晰？被拒方案是否经过真正考量？
> 影响章节是否诚实承认了权衡？引擎版本是否已标注？后知识截止日期的 API 风险
> 是否已标记？是否链接到其覆盖的 GDD 需求？返回 APPROVE、CONCERNS [具体空缺]，
> 或 REJECT [决策描述不充分或存在技术假设错误]。"

**裁定**：APPROVE / CONCERNS / REJECT

---

### TD-ENGINE-RISK — 引擎版本风险评审

**触发条件**：当架构决策涉及知识截止日期后的引擎 API 时，或在最终确定任何
特定引擎的实现方案之前

**需要传入的上下文**：
- 使用的具体 API 或功能
- 引擎版本及 LLM 知识截止日期（来自 `docs/engine-reference/[engine]/VERSION.md`）
- 破坏性变更或废弃 API 文档的相关摘录

**Prompt**：
> "对照版本参考文档评审此引擎 API 的使用。该 API 在 [引擎版本] 中是否存在？
> 自 LLM 知识截止日期以来，其签名、行为或命名空间是否发生变化？是否有已知的
> 废弃项或截止日期后的替代方案？返回 APPROVE（可按描述安全使用）、CONCERNS
> [实现前需验证]，或 REJECT [API 已变更——提供修正后的方案]。"

**裁定**：APPROVE / CONCERNS / REJECT

---

### TD-PHASE-GATE — 阶段过渡的技术就绪检查

**触发条件**：在 `/gate-check` 时始终触发——与 CD-PHASE-GATE 和 PR-PHASE-GATE 并行生成

**需要传入的上下文**：
- 目标阶段名称
- 架构文档路径（如存在）
- 引擎参考路径
- ADR 列表

**Prompt**：
> "从技术方向角度，评审当前项目状态是否满足进入 [目标阶段] 的就绪标准。
> 该阶段的架构是否健壮？所有高风险引擎领域是否已得到处理？性能预算是否
> 切实可行且有文档记录？基础层决策是否已足够完整以开始实现？返回 READY、
> CONCERNS [列表]，或 NOT READY [阻塞项]。"

**裁定**：READY / CONCERNS / NOT READY

## Tier 1 — 制作人门禁

Agent：`producer` | 模型层级：Opus | 领域：范围、时间线、依赖关系、生产风险

---

### PR-SCOPE — 范围与时间线验证

**触发条件**：范围层级定义完成后（头脑风暴第 6 阶段、quick-design，或任何
产出 MVP 定义和时间线估算的工作流）

**需要传入的上下文**：
- 完整愿景范围描述
- MVP 定义
- 时间线估算
- 团队规模（独立 / 小型团队等）
- 范围层级（时间不足时的交付计划）

**Prompt**：
> "评审这份范围估算。对于声明的团队规模，MVP 能否在声明的时间线内完成？
> 范围层级的顺序是否按风险正确排列——如果工作在某层级停止，该层级能否交付
> 可发布的产品？在时间压力下最可能的削减点是什么，它是优雅的备用方案还是
> 残缺的产品？返回 REALISTIC（范围与容量匹配）、OPTIMISTIC [建议的具体调整]，
> 或 UNREALISTIC [阻塞项——时间线或 MVP 必须修订]。"

**裁定**：REALISTIC / OPTIMISTIC / UNREALISTIC

---

### PR-SPRINT — Sprint 可行性评审

**触发条件**：最终确定 Sprint 计划前（`/sprint-plan`），以及 Sprint 中途发生
范围变更后

**需要传入的上下文**：

## 并行门禁协议

当工作流在同一检查点需要多位总监时（最常见于 `/gate-check`），同时生成所有 Agent：

```
并行生成（在等待任何结果之前先发出所有 Task 调用）：
1. creative-director  → 门禁 CD-PHASE-GATE
2. technical-director → 门禁 TD-PHASE-GATE
3. producer           → 门禁 PR-PHASE-GATE
4. art-director       → 门禁 AD-PHASE-GATE

收集全部四个裁定后，应用升级规则：
- 任何 NOT READY / REJECT → 总体裁定最低为 FAIL
- 任何 CONCERNS → 总体裁定最低为 CONCERNS
- 全部 READY / APPROVE → 可能通过 PASS（仍需满足产出物检查）
```

---

## 添加新门禁

当某个新技能或工作流需要新门禁时：

1. 分配门禁 ID：`[总监前缀]-[描述性标识]`
   - 前缀：`CD-` `TD-` `PR-` `LP-` `QL-` `ND-` `AD-`
   - 为新 Agent 添加新前缀：`AudioDirector` → `AU-`、`UX` → `UX-`
2. 在相应总监章节下添加该门禁，包含全部五个字段：
   触发条件、需要传入的上下文、Prompt、裁定，以及任何特殊处理说明
3. 在技能中仅通过 ID 引用——绝不将 prompt 文本复制到技能中
