# 上下文管理

上下文是 Claude Code 会话中最关键的资源，需主动管理。

## 文件持久化状态（首选策略）

**文件才是记忆，而非对话。** 对话内容是短暂的，随时可能被压缩或丢失。
磁盘上的文件在压缩和会话崩溃后仍然持续存在。

### 会话状态文件

将 `production/session-state/active.md` 作为动态检查点，在每个重要里程碑后更新：

- 设计章节已审批并写入文件
- 做出架构决策
- 到达实现里程碑
- 获得测试结果

状态文件应包含：当前任务、进度清单、已做决策、正在处理的文件，以及待解决的问题。

### 状态行块（仅限 Production 阶段及之后）

当项目处于 Production、Polish 或 Release 阶段时，在 `active.md` 中加入结构化状态块，
供状态行脚本解析：

```markdown
<!-- STATUS -->
Epic: Combat System
Feature: Melee Combat
Task: Implement hitbox detection
<!-- /STATUS -->
```

- 三个字段（Epic、Feature、Task）均为可选——按需填写
- 切换工作重心时更新此块
- 状态行以面包屑形式显示：`Combat System > Melee Combat > Hitboxes`
- 无活跃工作时清空或删除此块

发生任何中断（压缩、崩溃、`/clear`）后，优先读取状态文件。

### 增量文件写入

创建多章节文档（设计文档、架构文档、世界观条目）时：

1. 立即创建包含所有章节标题的骨架文件（正文留空）
2. 在对话中逐章讨论和起草
3. 每章审批通过后立即写入文件
4. 每章写入后更新会话状态文件
5. 某章写入文件后，之前关于该章的讨论可安全压缩——决策已记录在文件中

这样上下文窗口只需保存*当前章节*的讨论内容（约 3–5k tokens），
而非整个文档的对话历史（约 30–50k tokens）。

## 主动压缩

- 在上下文使用达 **~60–70%** 时主动压缩，而非等到达到上限后被动压缩
- 不相关任务之间或两次纠错失败后使用 **`/clear`**
- **自然压缩时机：** 写入章节到文件后、提交后、完成任务后、开始新话题前
- **聚焦压缩：** `/compact 专注于 [当前任务] — 第 1-3 节已写入文件，正在进行第 4 节`

## 各任务类型的上下文预算

- 轻量（读取/评审）：启动约 3k tokens
- 中等（实现功能）：约 8k tokens
- 重度（多系统重构）：约 15k tokens

## Subagent 委托

使用 subagent 进行研究和探索，保持主会话整洁。
Subagent 在独立上下文窗口中运行，只返回摘要：

- **使用 subagent** 的场景：跨多个文件调查、探索不熟悉的代码，
  或读取文件将消耗 >5k tokens 的研究工作
- **直接读取** 的场景：确切知道需要查看哪 1–2 个文件时
- Subagent 不继承对话历史——在提示中提供完整上下文

## 压缩指令

上下文被压缩时，在摘要中保留以下内容：

- 对 `production/session-state/active.md` 的引用（读取它以恢复状态）
- 本会话修改的文件列表及其用途
- 做出的任何架构决策及其依据
- 活跃 sprint 任务及当前状态
- Agent 调用及其结果（成功/失败/阻塞）
- 测试结果（通过/失败数量、具体失败项）
- 未解决的阻塞项或等待用户输入的问题
- 当前任务及所处步骤
- 当前文档哪些章节已写入文件、哪些仍在进行中

**压缩后：** 读取 `production/session-state/active.md` 及正在处理的文件以恢复完整上下文。
文件中保存着决策内容；对话历史是次要的。

## 会话崩溃后的恢复

如果会话死亡（"提示过长"）或开启新会话继续工作：

1. `session-start.sh` hook 会自动检测并预览 `active.md`
2. 读取完整状态文件获取上下文
3. 读取状态文件中列出的部分完成文件
4. 从下一个未完成的章节或任务继续
