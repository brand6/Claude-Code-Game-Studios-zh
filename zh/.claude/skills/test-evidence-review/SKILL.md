# 测试证据评审

`/smoke-check` 验证测试文件**存在**且**通过**。本技能更进一步——审查这些测试和证据文档的**质量**。
一个存在且通过的测试文件可能仍然留下关键行为未被覆盖。
一个存在的手动证据文档可能缺少关闭所需的签收。

**输出：** 摘要报告（在对话中）+ 可选的 `production/qa/evidence-review-[date].md`

**何时运行：**
- QA 移交签收前（`/team-qa` 阶段5）
- 任何测试质量存疑的故事
- 里程碑评审期间，用于逻辑和集成故事的质量审计

---

## 1. 解析参数

**模式：**
- `/test-evidence-review [story-path]` — 评审单个故事的证据
- `/test-evidence-review sprint` — 评审当前冲刺中所有故事
- `/test-evidence-review [system-name]` — 评审一个史诗/系统中的所有故事
- 无参数 — 询问范围："单个故事"、"当前冲刺"、"某个系统"

---

## 2. 加载范围内的故事

根据参数：

**单个故事**：直接读取故事文件。提取：故事类型、测试证据部分、故事 slug、系统名称。

**冲刺**：读取 `production/sprints/` 中最近修改的文件。从冲刺计划中提取故事文件路径列表，读取每个故事文件。

**系统**：Glob `production/epics/[system-name]/story-*.md`，读取每个。

对每个故事，收集：
- `Type:` 字段（逻辑 / 集成 / 视觉/手感 / UI / 配置/数据）
- `## Test Evidence` 部分——声明的预期测试文件路径或证据文档
- 故事 slug（来自文件名）
- 系统名称（来自目录路径）
- 验收标准列表（所有复选框项）

---

## 3. 定位证据文件

对每个故事，查找证据：

**逻辑故事**：Glob `tests/unit/[system]/[story-slug]_test.*`
  - 若未找到，也尝试：在 `tests/unit/[system]/` 中 Grep 包含故事 slug 的文件

**集成故事**：Glob `tests/integration/[system]/[story-slug]_test.*`
  - 同时检查 `production/session-logs/` 中提及该故事的游戏测试记录

**视觉/手感和 UI 故事**：Glob `production/qa/evidence/[story-slug]-evidence.*`

**配置/数据故事**：Glob `production/qa/smoke-*.md`（任意冒烟检查报告）

记录每个故事已找到（路径）或未找到（缺口）的内容。

---

## 4. 评审自动化测试质量（逻辑 / 集成）

对每个找到的测试文件，读取并评估：

### 断言覆盖率

统计不同断言的数量（包含 assert、expect、check、verify 或引擎特定断言模式的行）。低断言数量是质量信号——每个测试函数只有 1 个断言可能无法覆盖预期行为的范围。

阈值：
- **每个测试函数 3+ 个断言** → 正常
- **每个测试函数 1-2 个断言** → 记为可能偏薄
- **0 个断言**（测试存在但无断言）→ 标记为 BLOCKING——测试空跑通过，什么都没证明

### 边界情况覆盖率

对故事中包含数字、阈值或"当 X 发生时"条件的每个验收标准：检查测试函数名称或测试体是否引用了该具体情况。

启发式方法：
- 在测试文件中 Grep "zero"、"max"、"null"、"empty"、"min"、"invalid"、"boundary"、"edge"——出现任何一个均为正面信号
- 若故事有带具体边界的公式部分：检查测试是否在最小值/最大值上执行

### 命名质量

测试函数名称应描述：场景 + 预期结果。
模式：`test_[scenario]_[expected_outcome]`

将泛化命名的函数（`test_1`、`test_run`、`testBasic`）标记为**命名问题**——它们让失败更难诊断。

### 公式可追溯性

对于 GDD 有公式部分的逻辑故事：检查测试文件是否至少有一个测试，其名称或注释引用了公式名称或公式值。不提及公式名称就执行公式的测试，在公式变更时更难维护。

---

## 5. 评审手动证据质量（视觉/手感 / UI）

对每个找到的证据文档，读取并评估：

### 标准关联

证据文档应引用故事中的每个验收标准。检查：证据文档是否包含每个标准（或明确的改述）？缺失的标准意味着该标准从未被验证。

### 签收完整性

检查三条签收行（或等效字段）：
- 开发者签收
- 设计师/美术负责人签收（针对视觉/手感）
- QA 负责人签收

若有任何缺失或空白：标记为 INCOMPLETE——缺少所有必要签收时故事无法完全关闭。

### 截图/制品完整性

对于视觉/手感故事：检查证据文档中是否引用了截图文件路径。若已引用，Glob 确认它们存在。

对于 UI 故事：检查是否有演练序列（逐步交互日志）。

### 日期覆盖

证据文档应有日期。若日期早于故事最后一次重大变更（启发式：与冲刺计划中的冲刺开始日期比较），标记为 POTENTIALLY STALE——证据可能未覆盖最终实现。

---

## 6. 生成评审报告

对每个故事，指定结论：

| 结论 | 含义 |
|------|------|
| **ADEQUATE** | 测试/证据存在，通过质量检查，所有标准已覆盖 |
| **INCOMPLETE** | 测试/证据存在但有质量缺口（断言偏薄、签收缺失） |
| **MISSING** | 未找到需要它的故事类型的测试或证据 |

总体冲刺/系统结论是所有故事结论中最差的。

```markdown
## 测试证据评审

> **日期**：[date]
> **范围**：[单个故事路径 | 冲刺 [N] | [系统名称]]
> **评审故事数**：[N]
> **总体结论**：ADEQUATE / INCOMPLETE / MISSING

---

### 故事逐项结果

#### [故事标题] — [类型] — [ADEQUATE/INCOMPLETE/MISSING]

**测试/证据路径**：`[path]`（已找到）/ （未找到）

**自动化测试质量** *（仅限逻辑/集成）*：
- 断言覆盖率：[平均每函数 N 个]——[充足 / 偏薄 / 无]
- 边界情况：[已覆盖 / 部分覆盖 / 未找到]
- 命名：[一致 / 标记了 [N] 个泛化名称]
- 公式可追溯性：[是 / 否——测试中未引用公式名称]

**手动证据质量** *（仅限视觉/手感/UI）*：
- 标准关联：[N/M 个标准已引用]
- 签收：[开发者 ✓ | 设计师 ✗ | QA 负责人 ✗]
- 制品：[截图已存在 / 缺失 / 不适用]
- 新鲜度：[日期 [date]——当前 / 可能过时]

**问题**：
- BLOCKING：[描述] *（阻止故事完成）*
- ADVISORY：[描述] *（应在发布前修复）*

---

### 摘要

| 故事 | 类型 | 结论 | 问题 |
|------|------|------|------|
| [title] | 逻辑 | ADEQUATE | 无 |
| [title] | 集成 | INCOMPLETE | 断言偏薄（平均 1.2/函数） |
| [title] | 视觉/手感 | INCOMPLETE | QA 负责人签收缺失 |
| [title] | 逻辑 | MISSING | 未找到测试文件 |

**BLOCKING 项**（故事关闭前必须解决）：[N]
**ADVISORY 项**（发布前应处理）：[N]
```

---

## 7. 写入输出（可选）

在对话中呈现报告。

询问："我可以将此测试证据评审写入 `production/qa/evidence-review-[date].md` 吗？"

这是可选的——报告单独使用也很有价值。只有在用户想要持久记录时才写入。

报告之后：

- 对于 BLOCKING 项："这些必须在 `/story-done` 将故事标记为完成之前解决。现在处理其中任何一项吗？"
- 对于断言偏薄："考虑运行 `/test-helpers [system]` 查看常见情况的断言模式脚手架。"
- 对于缺少签收："需要来自 [角色] 的手动签收。将 `[evidence-path]` 分享给他们完成签收。"

结论：**COMPLETE** — 证据评审完成。若发现 BLOCKING 项则使用 CONCERNS。

---

## 协作协议

- **报告质量问题，不修复** — 本技能读取并评估；不修改测试文件或证据文档
- **ADEQUATE 意味着足够发货，不是完美** — 避免对功能正常且覆盖充分的测试吹毛求疵
- **BLOCKING 与 ADVISORY 的区分很重要** — 仅当缺口使某个故事标准真正未经验证时才标记 BLOCKING
- **写入前询问** — 报告文件是可选的；写入前始终确认
