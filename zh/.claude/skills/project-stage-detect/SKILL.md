# 项目阶段检测

本技能扫描你的项目以确定其当前开发阶段、制品的完整性和需要关注的缺口。特别适用于：
- 加入现有项目时
- 向代码库上手时
- 在里程碑前检查缺少什么
- 了解"我们在哪里？"

---

## 工作流

### 1. 扫描关键目录

分析项目结构和内容：

**设计文档**（`design/`）：
- 统计 `design/gdd/*.md` 中的 GDD 文件数量
- 检查 game-concept.md、game-pillars.md、systems-index.md 是否存在
- 若 systems-index.md 存在，统计总系统数与已设计系统数
- 分析完整性（概述、详细设计、边界情况等）
- 统计 `design/narrative/` 中的叙事文档
- 统计 `design/levels/` 中的关卡设计

**源代码**（`src/`）：
- 统计源文件数量（语言无关）
- 识别主要系统（包含 5 个以上文件的目录）
- 检查 core/、gameplay/、ai/、networking/、ui/ 目录
- 估计代码行数（粗略规模）

**生产制品**（`production/`）：
- 检查活跃迭代计划
- 查找里程碑定义
- 寻找路线图文档

**原型**（`prototypes/`）：
- 统计原型目录数量
- 检查 README（已记录 vs 未记录）
- 评估原型是存档还是活跃状态

**架构文档**（`docs/architecture/`）：
- 统计 ADR（架构决策记录）数量
- 检查概述/索引文档

**测试**（`tests/`）：
- 统计测试文件数量
- 估计测试覆盖率（粗略启发式）

### 2. 分类项目阶段

根据扫描的制品确定阶段。首先检查 `production/stage.txt` —— 若存在，使用其值（来自 `/gate-check` 的明确覆盖）。否则，使用以下启发式方法自动检测（从最高阶段向后检查）：

| 阶段 | 指标 |
|------|------|
| **概念** | 无游戏概念文档，处于头脑风暴阶段 |
| **系统设计** | 游戏概念存在，系统索引缺失或不完整 |
| **技术设置** | 系统索引存在，引擎未配置 |
| **预制作** | 引擎已配置，`src/` 中源文件少于 10 个 |
| **生产** | `src/` 中有 10 个以上源文件，正在活跃开发 |
| **打磨** | 仅显式（由 `/gate-check` 生产→打磨门禁设置） |
| **发布** | 仅显式（由 `/gate-check` 打磨→发布门禁设置） |

### 3. 协作式缺口识别

**不要**只列出缺失文件。而是**提问澄清性问题**：

- "我看到战斗代码（`src/gameplay/combat/`）但没有 `design/gdd/combat-system.md`。这是先原型化的，还是我们应该进行逆向文档化？"
- "你有 15 个 ADR 但没有架构概述。我应该创建一个来帮助新贡献者吗？"
- "`production/` 中没有迭代计划。你是在其他地方跟踪工作吗（Jira、Trello 等）？"
- "我找到了游戏概念但没有系统索引。你是否已经将概念分解为各个系统，还是我们应该运行 `/map-systems`？"
- "原型目录中有 3 个项目没有 README。这些是实验，还是需要文档化？"

### 4. 生成阶段报告

使用模板：`.claude/docs/templates/project-stage-report.md`

**报告结构**：
```markdown
# 项目阶段分析

**日期**：[日期]
**阶段**：[概念/系统设计/技术设置/预制作/生产/打磨/发布]
**阶段置信度**：[PASS —— 清晰检测 / CONCERNS —— 模糊信号 / FAIL —— 关键缺口阻塞进度]

## 完整性概览
- 设计：[X%]（[N] 个文档，[缺口]）
- 代码：[X%]（[N] 个文件，[系统]）
- 架构：[X%]（[N] 个 ADR，[缺口]）
- 生产：[X%]（[状态]）
- 测试：[X%]（[覆盖率估计]）

## 已识别的缺口
1. [缺口描述 + 澄清问题]
2. [缺口描述 + 澄清问题]

## 推荐的后续步骤
[根据阶段和角色优先级排序的列表]
```

### 5. 角色过滤推荐（可选）

若用户提供了角色参数（例如 `/project-stage-detect programmer`）：

**程序员**：
- 侧重于架构文档、测试覆盖率、缺失的 ADR
- 代码与文档的缺口

**设计师**：
- 侧重于 GDD 完整性、缺失的设计章节
- 原型文档

**制作人**：
- 侧重于迭代计划、里程碑跟踪、路线图
- 跨团队协调文档

**通用**（无角色）：
- 所有缺口的整体视图
- 跨域的最高优先级项目

### 6. 写入前请求批准

**协作协议**：
```
我已分析了你的项目。以下是我发现的内容：

[显示摘要]

已识别的缺口：
1. [缺口 1 + 问题]
2. [缺口 2 + 问题]

推荐的后续步骤：
- [优先级 1]
- [优先级 2]
- [优先级 3]

我可以将完整阶段分析写入 production/project-stage-report.md 吗？
```

等待用户批准后再创建文件。

## 后续行动

生成报告后，建议相关的后续步骤：

- **概念存在但无系统索引？** → `/map-systems` 将概念分解为系统
- **缺少设计文档？** → `/reverse-document design src/[system]`
- **缺少架构文档？** → `/architecture-decision` 或 `/reverse-document architecture`
- **原型需要文档？** → `/reverse-document concept prototypes/[name]`
- **无迭代计划？** → `/sprint-plan`
- **接近里程碑？** → `/milestone-review`

---

## 协作协议

本技能遵循协作设计原则：

1. **先提问**：对缺口进行询问，不假设
2. **呈现选项**："我应该创建 X，还是它在其他地方跟踪？"
3. **用户决定**：等待方向
4. **展示草稿**：显示报告摘要
5. **获得批准**："我可以写入 production/project-stage-report.md 吗？"

**绝不**静默写入文件。**始终**展示发现并在创建制品之前询问。
