# 实现用户故事

本技能衔接规划与代码。它完整读取故事文件，组装程序员所需的所有上下文，路由到正确的专业智能体，并将实现推进到完成 —— 包括编写测试。

**每个故事的循环：**
```
/qa-plan sprint           ← 迭代开始前定义测试需求
/story-readiness [path]   ← 开始前验证
/dev-story [path]         ← 实现它（本技能）
/code-review [files]      ← 审查
/story-done [path]        ← 验证并关闭
```

**所有迭代故事完成后：** 在推进项目阶段之前，运行 `/team-qa sprint` 执行完整 QA 周期并获得签字裁定结果。

**输出：** 项目 `src/` 和 `tests/` 目录中的源代码 + 测试文件。

---

## 阶段 1：找到故事

**若提供了路径**：直接读取该文件。

**若无参数**：检查 `production/session-state/active.md` 中的活跃故事。若找到，确认："继续处理 [故事标题] —— 这样对吗？"若未找到，询问："我们要实现哪个故事？"使用 Glob 找到 `production/epics/**/*.md`，列出状态为 Ready 的故事。

---

## 阶段 2：加载完整上下文

**加载任何上下文之前，验证必需文件是否存在。** 从故事的"ADR 治理实现"字段提取 ADR 路径，然后检查：

| 文件 | 路径 | 若缺失 |
|------|------|--------|
| TR 注册表 | `docs/architecture/tr-registry.yaml` | **停止** —— "未找到 TR 注册表。运行 `/create-epics` 生成它。" |
| 治理 ADR | 来自故事 ADR 字段的路径 | **停止** —— "未找到 ADR 文件 [path]。运行 `/architecture-decision` 创建它，或更正故事 ADR 字段中的文件名。" |
| 控制清单 | `docs/architecture/control-manifest.md` | **警告并继续** —— "未找到控制清单 —— 无法检查层级规则。运行 `/create-control-manifest`。" |

若 TR 注册表或治理 ADR 缺失，在会话状态中将故事状态设置为 **BLOCKED**，不生成任何程序员智能体。

同时读取以下所有内容 —— 这些是独立读取。在所有上下文加载完毕之前不要开始实现：

### 故事文件
提取并保存：
- **故事标题、ID、层级、类型**（逻辑 / 集成 / 视觉/感受 / UI / 配置/数据）
- **TR-ID** —— GDD 需求标识符
- **治理 ADR** 引用
- **清单版本** 嵌入在故事头部
- **验收标准** —— 每个勾选项，逐字记录
- **实现注意事项** —— 故事中的 ADR 指南章节
- **超出范围** 边界
- **测试证据** —— 所需测试文件路径
- **依赖项** —— 本故事之前必须完成的内容

### TR 注册表
读取 `docs/architecture/tr-registry.yaml`。查找故事的 TR-ID。
读取当前 `requirement` 文本 —— 这是 GDD 当前要求什么的事实来源。不要依赖故事文件中的任何内联文本（可能已过时）。

### 治理 ADR
读取 `docs/architecture/[adr-file].md`。提取：
- 完整的决策章节
- 实现指南章节（这是程序员遵循的内容）
- 引擎兼容性章节（训练截止后的 API、已知风险）
- ADR 依赖章节

### 控制清单
读取 `docs/architecture/control-manifest.md`。提取该故事层级的规则：
- 必需模式
- 禁止模式
- 性能护栏

检查：故事嵌入的清单版本是否与当前清单头部日期匹配？
若不同，在继续前使用 `AskUserQuestion`：
- 提示："故事是针对清单 v[故事日期] 编写的。当前清单为 v[当前日期]。新规则可能适用。如何继续？"
- 选项：
  - `[A] 更新故事清单版本并按当前规则实现（推荐）`
  - `[B] 按旧规则实现 —— 我接受不合规风险`
  - `[C] 停在这里 —— 我想先查看清单差异`

若 [A]：在生成程序员之前编辑故事文件的 `Manifest Version:` 字段为当前清单日期。然后仔细阅读清单中的新规则。
若 [B]：无论如何仔细阅读清单中的新规则，并在阶段 6 摘要的"偏差"下注明版本不匹配。
若 [C]：停止。不生成任何智能体。让用户审查后重新运行 `/dev-story`。

### 依赖项验证

从故事文件提取**依赖项**列表后，验证每一项：

1. 使用 Glob 找到 `production/epics/**/*.md`，定位每个依赖故事文件。
2. 读取其 `Status:` 字段。
3. 若任何依赖项的状态不是 `Complete` 或 `Done`：
   - 使用 `AskUserQuestion`：
     - 提示："故事'[当前故事]'依赖于'[依赖标题]'，当前状态为 [状态]，并非 Complete。如何继续？"
     - 选项：
       - `[A] 无论如何继续 —— 我接受依赖风险`
       - `[B] 停止 —— 我先完成依赖项`
       - `[C] 依赖项已完成但状态未更新 —— 标记为 Complete 并继续`
   - 若 [B]：在会话状态中将故事状态设置为 **BLOCKED** 并停止。
   - 若 [C]：在继续前询问"我可以将 [依赖路径] 状态更新为 Complete 吗？"
   - 若 [A]：在阶段 6 摘要的"偏差"下注明："以不完整的依赖项实现：[依赖标题] —— [状态]。"

若找不到依赖文件：警告"未找到依赖故事：[path]。请验证路径或创建故事文件。"

### 引擎参考
读取 `.claude/docs/technical-preferences.md`：
- `Engine:` 值 —— 决定使用哪些程序员智能体
- 命名约定（类名、文件名、信号/事件名）
- 性能预算（帧预算、内存上限）
- 禁止模式

## 阶段 3：路由到正确的程序员

根据故事的**层级**、**类型**和**系统名称**，确定通过 Task 生成哪个专业智能体。

**配置/数据故事 —— 完全跳过智能体生成：**
若故事类型为 `Config/Data`，不需要程序员智能体或引擎专家。直接跳到阶段 4（配置/数据说明）。

### 主要智能体路由表

| 故事上下文 | 主要智能体 |
|-----------|-----------|
| Foundation 层 —— 任何类型 | `engine-programmer` |
| 任何层 —— 类型：UI | `ui-programmer` |
| 任何层 —— 类型：视觉/感受 | `gameplay-programmer`（实现） |
| Core 或 Feature —— 游戏机制 | `gameplay-programmer` |
| Core 或 Feature —— AI 行为、寻路 | `ai-programmer` |
| Core 或 Feature —— 网络、复制 | `network-programmer` |
| 配置/数据 —— 无代码 | 不需要智能体（见阶段 4 配置说明） |

### 引擎专家 —— 代码故事始终生成为次要智能体

从 `.claude/docs/technical-preferences.md` 的 `Engine Specialists` 部分读取已配置的主要专家。当故事涉及引擎特定 API、模式，或 ADR 具有 HIGH 引擎风险时，与主要智能体一起生成。

| 引擎 | 可用专家智能体 |
|------|-------------|
| Godot 4 | `godot-specialist`、`godot-gdscript-specialist`、`godot-shader-specialist` |
| Unity | `unity-specialist`、`unity-ui-specialist`、`unity-shader-specialist` |
| Unreal Engine | `unreal-specialist`、`ue-gas-specialist`、`ue-blueprint-specialist`、`ue-umg-specialist`、`ue-replication-specialist` |

**当引擎风险为 HIGH 时**（来自 ADR 或 VERSION.md）：始终生成引擎专家，即使是非引擎相关的故事。高风险意味着 ADR 记录了对训练截止后引擎 API 的假设，需要专家验证。

## 阶段 4：实现

通过 Task 生成选定的程序员智能体，提供完整上下文包：

提供给智能体：
1. 完整故事文件内容
2. 当前 GDD 需求文本（来自 TR 注册表）
3. ADR 决策 + 实现指南（逐字 —— 不要摘要）
4. 该层级的控制清单规则
5. 引擎命名约定和性能预算
6. ADR 引擎兼容性章节的任何引擎特定注意事项
7. 必须创建的测试文件路径
8. 明确指令：**实现本故事并编写测试**

智能体应：
- 按照 ADR 指南在 `src/` 中创建或修改文件
- 遵守控制清单中的所有必需和禁止模式
- 保持在故事的"超出范围"边界内（不触碰不相关的文件）
- 编写有清晰文档注释的公共 API

### 配置/数据故事（不需要智能体）

对于类型为配置/数据的故事，不需要程序员智能体。实现是编辑数据文件。读取故事的验收标准，直接对数据文件进行指定的更改。记录更改了哪些值以及从什么改为什么。

### 视觉/感受故事

生成 `gameplay-programmer` 实现代码/动画调用。注意视觉/感受验收标准无法自动验证 —— "感觉对吗？"检查发生在 `/story-done` 中的手动确认。

---

## 阶段 5：编写测试

对于**逻辑**和**集成**故事，测试必须作为本次实现的一部分编写 —— 不推迟到以后。

提醒程序员智能体：

> "本故事的测试文件是必须的，路径为：`[来自测试证据章节的路径]`。
> 没有它，故事无法通过 `/story-done` 关闭。实现时同步编写测试，而非之后补充。"

测试要求（来自 coding-standards.md）：
- 文件名：`[system]_[feature]_test.[ext]`
- 函数名：`test_[scenario]_[expected_outcome]`
- 每个验收标准至少有一个测试函数覆盖
- 无随机种子、无时间相关断言、无外部 I/O
- 测试 GDD 公式章节中的公式边界

对于**视觉/感受**和 **UI** 故事：无自动测试。提醒智能体在实现摘要中注明所需的手动证据：
"证据文档需要在 `production/qa/evidence/[slug]-evidence.md`。"

对于**配置/数据**故事：无测试文件。冒烟测试将作为证据。

## 阶段 6：收集并汇总

程序员智能体完成后，收集：

- 已创建或修改的文件（包含路径）
- 已创建的测试文件（路径和编写的测试函数数量）
- 超出故事"超出范围"边界的任何偏差（标记这些）
- 智能体提出的任何问题或阻塞项
- 专家标记的任何引擎特定风险

呈现简洁的实现摘要：

```
## 实现完成：[故事标题]

**已修改文件**：
- `src/[path]` —— 已创建/修改（[简短描述]）
- `tests/[path]` —— 测试文件（[N] 个测试函数）

**已覆盖验收标准**：
- [x] [标准] —— 在 [file:function] 中实现
- [x] [标准] —— 由测试 [test_name] 覆盖
- [ ] [标准] —— 已延迟：需要游戏测试（视觉/感受）

**范围偏差**：[无] 或 [列出超出故事范围触碰的文件]
**引擎风险标记**：[无] 或 [专家发现]
**阻塞项**：[无] 或 [描述]

准备进行：`/code-review [file1] [file2]` 然后 `/story-done [story-path]`
```

## 阶段 7：更新会话状态

静默追加到 `production/session-state/active.md`：

```
## 会话摘录 —— /dev-story [日期]
- 故事：[story-path] —— [故事标题]
- 已修改文件：[逗号分隔列表]
- 已编写测试：[path，或"无 —— 视觉/感受/配置故事"]
- 阻塞项：[无，或描述]
- 下一步：/code-review [files] 然后 /story-done [story-path]
```

若 `active.md` 不存在则创建它。确认："会话状态已更新。"

## 错误恢复协议

若任何通过 Task 生成的智能体返回 BLOCKED、报错或无法完成：

1. **立即提示**：在继续依赖阶段之前向用户报告"[AgentName]：BLOCKED —— [原因]"
2. **评估依赖关系**：检查被阻塞智能体的输出是否被后续阶段所需。若是，不要在没有用户输入的情况下越过该依赖点。
3. **通过 AskUserQuestion 提供选项**：
   - 跳过此智能体并在最终报告中记录差距
   - 以更窄的范围重试
   - 停在这里先解决阻塞项
4. **始终生成部分报告** —— 输出已完成的内容。不要因为一个智能体阻塞就丢弃工作。

常见阻塞项：
- 输入文件缺失（故事未找到、GDD 缺失）→ 重定向到创建它的技能
- ADR 状态为 Proposed → 不要实现；先运行 `/architecture-decision`
- 范围太大 → 通过 `/create-stories` 拆分为两个故事
- ADR 和故事之间的冲突指令 → 提示冲突，不要猜测
- 清单版本不匹配 → 向用户展示差异，询问是否按旧规则继续或先更新故事

## 协作协议

- **文件写入被委托** —— 所有源代码、测试文件和证据文档由通过 Task 生成的子智能体编写。每个子智能体单独执行"我可以写入 [path] 吗？"协议。本编排器不直接写文件。
- **先加载再实现** —— 在所有上下文加载完毕之前不要开始编码（故事、TR-ID、ADR、清单、引擎偏好）。
- **ADR 是法律** —— 实现必须遵循 ADR 的实现指南。若指南与看起来"更好"的方法冲突，在摘要中标记而非静默偏离。
- **保持在范围内** —— "超出范围"章节是合约。若实现故事需要触碰超出范围的文件，停止并提示："实现 [标准] 需要修改 [file]，这超出了范围。是否继续或创建单独的故事？"
- **逻辑/集成故事的测试不可选** —— 没有测试文件存在，不标记实现为完成
- **视觉/感受标准被延迟，而非跳过** —— 在摘要中标记为 DEFERRED；它们将在 `/story-done` 中手动验证
- **大型结构决策前先询问** —— 若故事需要 ADR 未涵盖的架构模式，在实现前提示："ADR 未说明如何处理 [情况]。我的计划是 [X]。继续吗？"

## 建议后续步骤

- 运行 `/code-review [file1] [file2]` 在关闭故事前审查实现
- 运行 `/story-done [story-path]` 验证验收标准并将故事标记为完成
- 所有迭代故事完成后：在推进项目阶段之前运行 `/team-qa sprint` 执行完整 QA 周期
