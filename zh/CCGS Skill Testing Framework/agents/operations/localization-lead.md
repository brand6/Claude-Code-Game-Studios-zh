# Agent 测试规格：localization-lead

## Agent 概述
- **职责领域**：国际化（i18n）架构、字符串提取工作流与工具配置、语言测试方法论、翻译流水线设计（提取 → TMS → 导入）、字符串质量标准、地区特定格式规则（复数形式、RTL、日期/数字格式）
- **不负责**：游戏叙事内容与对话写作（writer）、i18n 调用的代码实现（gameplay-programmer）、翻译工作本身（外部翻译人员）
- **模型层级**：Sonnet
- **关卡 ID**：无；当流水线架构决策影响构建系统时，上报 technical-director

---

## 静态断言（结构检查）

- [ ] `description:` 字段存在且领域明确（引用 i18n、字符串提取、地区流水线、本地化）
- [ ] `allowed-tools:` 列表与角色职责匹配（可读写本地化配置、流水线文档、字符串表；不含游戏源码编辑或部署工具）
- [ ] 模型层级为 Sonnet（专员的默认层级）
- [ ] Agent 定义未主张对叙事内容、游戏代码实现或翻译质量拥有权

## 测试用例

### 用例 1：领域内请求——为 Unity 项目设置字符串提取流水线
**输入**："为我们的 Unity 游戏设置字符串提取流水线。我们需要将所有可本地化的字符串整理成翻译人员可用的格式。"
**预期行为**：
- 产出具体的提取配置，涵盖：要提取哪些字符串类型（UI 标签、对话、物品描述——不含调试字符串）、使用什么工具（如 Unity Localization 包的字符串表，或针对特定组件类型的自定义提取脚本），以及输出格式（CSV、XLIFF 或 TMX——注明哪些格式与 Crowdin 或 Lokalise 等常见 TMS 工具兼容）
- 指定文件夹结构：如 `assets/localization/en/` 作为源语言，`assets/localization/{locale}/` 用于译文
- 注明字符串键必须稳定（不使用基于索引的键）——键名变更会破坏所有现有翻译
- 不产出 i18n 实现的 Unity C# 代码——标注为 [待程序员实现]

### 用例 2：领域外请求——翻译游戏对话
**输入**："将以下英文对话翻译成法文：'Well met, traveler. The road ahead is treacherous.'"
**预期行为**：
- 不产出法文翻译
- 明确声明："本地化负责人负责流水线、质量标准和工作流；实际翻译工作由人工翻译或经授权的翻译供应商完成——我不是翻译员"
- 可选：注明翻译人员需要什么信息：上下文（谁在说话、对谁说、游戏类型/语调）、若有的话的字符限制、词汇表术语（如"traveler"有游戏专属译法）

### 用例 3：领域边界——俄语地区文件缺少复数形式
**输入**："我们的俄语地区文件仅有物品数量字符串的单数形式。俄语需要多种复数形式（1个物品、2-4个物品、5+个物品使用不同形式）。"
**预期行为**：
- 识别这是一个地区特定复数形式缺失问题：俄语有3个复数类别（one、few、many），符合 CLDR/Unicode 复数规则——单一字符串不够用
- 标记为本地化质量 bug，而非轻微的风格问题——错误的复数形式在语法上是错误的，玩家可以看到
- 推荐修复方案：将字符串提取格式更新为支持 CLDR 复数类别（one/few/many/other），并通知翻译供应商俄语字符串需要所有复数形式
- 注明流水线中其他同样需要复数形式支持的语言（如波兰语、捷克语、阿拉伯语）
- 不建议使用数字阈值变通方案替代正规的 CLDR 复数支持

### 用例 4：两个系统之间的字符串键命名冲突
**输入**："我们的 UI 系统对相同概念使用 'button_confirm' 和 'button_cancel'，而对话系统使用 'confirm' 和 'cancel'。翻译人员对该用哪个感到困惑。"
**预期行为**：
- 识别冲突：两个系统对语义相同的字符串使用不同的键命名规范，造成重复翻译工作和翻译人员困惑
- 产出命名规范解决方案：带领域前缀的键，使用一致分隔符（如 `ui.button.confirm`、`ui.button.cancel`）——所有系统对共享概念使用同一个键
- 建议共享 UI 原语（确认、取消、返回、确定）在共享命名空间中使用单一规范键，供两个系统引用
- 提供迁移路径：将旧键映射到新键，更新两个系统中的所有字符串引用，在一个发布周期后废弃旧键
- 不建议为同一概念维护两个独立的键

### 用例 5：上下文传递——流水线适配 RTL 语言
**上下文输入**：目标地区包括英语（en）、法语（fr）、德语（de）、阿拉伯语（ar）、希伯来语（he）。
**输入**："为本项目设计本地化流水线。"
**预期行为**：
- 识别阿拉伯语和希伯来语为从右至左（RTL）语言——明确将其作为流水线需求提出
- 将流水线设计为包含：RTL 文本渲染支持（标注给程序员：UI 必须支持 RTL 布局镜像）、字符串表中的双向（bidi）文本处理、RTL 布局的地区专属测试清单条目
- 不在指定了 RTL 地区的情况下，设计仅考虑 LTR 语言的流水线
- 注明阿拉伯语还需要不同的复数形式结构（CLDR 中有6个复数类别）——标注给翻译供应商
- 输出在流水线架构中包含所有五个地区，而非只有默认语言（en）

## 协议合规

- [ ] 保持在声明领域内（流水线、提取、字符串质量、地区格式、i18n 架构）
- [ ] 不产出翻译——将翻译工作重定向给人工翻译/供应商
- [ ] 将地区特定缺失问题（复数形式、RTL）标记为需要流水线变更的质量 bug
- [ ] 在命名冲突出现时产出统一的键命名规范——不接受双重规范并存
- [ ] 将所有上下文提供的目标地区（包括 RTL 语言）纳入流水线设计

---

## 覆盖说明
- 用例 3（复数形式）和用例 5（RTL）是地区正确性测试——这些问题影响非英语市场的上线质量
- 用例 4（键命名冲突）是流水线规范测试——键重复会造成持续的翻译人员困惑和额外成本
- 用例 5 需要上下文中包含目标地区列表；若未提供，Agent 应在设计流水线前主动询问
- 无自动化运行程序；通过人工审阅或 `/skill-test` 进行测试
