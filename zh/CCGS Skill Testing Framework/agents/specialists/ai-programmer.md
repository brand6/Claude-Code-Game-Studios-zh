# Agent 测试规格：ai-programmer

## Agent 概述
职责领域：NPC 行为、状态机、寻路、感知系统与 AI 决策。
不负责：玩家机制（gameplay-programmer）、渲染或引擎内部（engine-programmer）。
模型层级：Sonnet（默认）。
未分配关卡 ID。

---

## 静态断言（结构检查）

- [ ] `description:` 字段存在且领域明确（引用 NPC 行为 / AI 系统）
- [ ] `allowed-tools:` 列表包含 Read、Write、Edit、Bash、Glob、Grep
- [ ] 模型层级为 Sonnet（专员的默认层级）
- [ ] Agent 定义未主张对玩家机制或引擎渲染拥有权

---

## 测试用例

### 用例 1：领域内请求——合适的输出
**输入**："为守卫 NPC 实现巡逻-警戒行为树：在路点间巡逻，10单位内检测到玩家后进入警戒状态并追击。"
**预期行为**：
- 产出行为树规格（节点：Selector、Sequence、叶节点动作）及对应代码骨架
- 定义清晰命名的状态：巡逻（Patrol）、警戒（Alert）、追击（Pursue）
- 将感知/检测检查作为条件节点，而非嵌入移动代码中
- 路点为数据驱动（作为资源或导出变量传入），不硬编码位置
- 输出包含公开 API 的文档注释

### 用例 2：领域外请求——正确重定向
**输入**："实现 WASD 移动和冲刺技能的玩家输入处理。"
**预期行为**：
- 不产出玩家输入或移动代码
- 明确声明这超出其职责范围（玩家机制属于 gameplay-programmer）
- 将请求重定向给 `gameplay-programmer`
- 可注明：一旦玩家位置通过 API 可用，AI 感知系统即可引用

### 用例 3：跨领域协调——关卡限制
**输入**："为仓库关卡设计寻路，但该关卡有窄走廊导致导航网格混乱。"
**预期行为**：
- 不单方面修改关卡布局或导航网格资产
- 与 `level-designer` 协调，明确导航网格需求和走廊尺寸
- 根据关卡几何形状，提出有条件的寻路方案（如带 Agent 半径调整的导航网格、流场）
- 清楚记录假设并标出阻塞项

### 用例 4：性能上报——自定义数据结构
**输入**："寻路优先级队列是性能瓶颈；需要一个自定义二叉堆实现来提升性能。"
**预期行为**：
- 认识到底层的、与引擎集成的数据结构属于 engine-programmer 的职责范围
- 向 `engine-programmer` 上报，并清晰描述瓶颈及所需接口
- 可提供算法规格（二叉堆接口、预期操作）以指导 engine-programmer
- 若实现需要引擎内存管理，不单方面实现底层结构

### 用例 5：上下文传递——使用关卡布局进行寻路设计
**输入**：上下文中提供了关卡布局文档，显示两处要道：(12, 0) 处的门道和 (40, 5) 处的桥梁。请求："为本关卡中的敌人设计巡逻路线和威胁应对方案。"
**预期行为**：
- 引用上下文中提供的具体要道坐标
- 设计利用要道作为战术位置的巡逻路线
- 指定警戒状态切换逻辑，使 NPC 在追击时朝已识别的要道集结
- 不凭空发明布局文档中不存在的几何信息

---

## 协议合规

- [ ] 保持在声明领域内（NPC 行为、寻路、感知、状态机）
- [ ] 将领域外请求重定向给正确的 Agent（gameplay-programmer、engine-programmer、level-designer）
- [ ] 返回结构化产出（行为树规格、状态机图、代码骨架）
- [ ] 未经明确授权，不修改玩家机制文件
- [ ] 将性能关键的底层结构上报给 engine-programmer
- [ ] 使用数据驱动的 NPC 配置（路点、检测半径），不硬编码值

---

## 覆盖说明
- 行为树输出（用例 1）应通过 `tests/unit/ai/` 中的单元测试验证
- 关卡布局上下文（用例 5）验证 Agent 是否读取并应用所提供的文档，而非自行发明
- 性能上报（用例 4）确认 Agent 能识别 engine-programmer 的职责边界
