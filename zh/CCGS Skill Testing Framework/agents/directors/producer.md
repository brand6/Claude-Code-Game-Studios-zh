# Agent 测试规格：producer

## Agent 摘要
**拥有领域：** 范围管理、冲刺规划验证、里程碑跟踪、史诗优先级排定、生产阶段关卡。
**不拥有：** 游戏设计决策（creative-director / game-designer）、技术架构（technical-director）、创意方向。
**模型层级：** Opus（多文档综合、高风险阶段关卡裁决）。
**处理的关卡 ID：** PR-SCOPE、PR-SPRINT、PR-MILESTONE、PR-EPIC、PR-PHASE-GATE。

---

## 静态断言（结构性）

通过读取 agent 的 `.claude/agents/producer.md` frontmatter 验证：

- [ ] `description:` 字段存在且针对特定领域（引用范围、冲刺、里程碑、生产 — 而非通用描述）
- [ ] `allowed-tools:` 列表主要以读取为主；仅在冲刺/里程碑文件需要解析时包含 Bash
- [ ] 模型层级为 `claude-opus-4-6`，符合 coordination-rules.md（具有关卡综合职能的 directors = Opus）
- [ ] Agent 定义不声明对设计决策或技术架构的权限

## 测试用例

### 用例 1：领域内请求 — 适当的输出格式
**场景：** 第 7 冲刺的冲刺计划提交审查。计划包含 4 名团队成员在 2 周内完成的 12 个故事点。过去 3 个冲刺的历史速度平均为 11.5 个故事点。请求标记为 PR-SPRINT。
**预期：** 返回 `PR-SPRINT: REALISTIC`，附理由说明计划在历史速度一个标准差以内，且容量与规划匹配。
**断言：**
- [ ] 裁决恰好为 REALISTIC / CONCERNS / UNREALISTIC 之一
- [ ] 裁决令牌格式为 `PR-SPRINT: REALISTIC`
- [ ] 理由引用具体的故事点数和历史速度数字
- [ ] 输出保持在生产范围内 — 不评论故事设计是否合理或技术是否可行

### 用例 2：领域外请求 — 重定向或升级
**场景：** 团队成员请求 producer 评估游戏的"基于重量的库存"机制是否有趣且有吸引力。
**预期：** Agent 拒绝评估游戏手感，并重定向至 game-designer 或 creative-director。
**断言：**
- [ ] 不对机制的设计质量做出任何约束性评估
- [ ] 明确指明 `game-designer` 或 `creative-director` 为正确的处理方
- [ ] 可以指出该机制的范围是否有生产影响（例如对其他系统的依赖），但将所有设计评估交由对方

### 用例 3：关卡裁决 — 正确的裁决词汇
**场景：** 一个新功能提案在一个仅计划两个系统的里程碑中增加了三个新系统（合成、天气和派系声望）。这些新增内容均未出现在当前里程碑计划中。请求标记为 PR-SCOPE。
**预期：** 返回 `PR-SCOPE: CONCERNS`，具体指出三个计划外系统及其未出现在里程碑范围文档中的情况。
**断言：**
- [ ] 裁决恰好为 REALISTIC / CONCERNS / UNREALISTIC 之一 — 而非自由文本
- [ ] 裁决令牌格式为 `PR-SCOPE: CONCERNS`
- [ ] 理由指出被超范围添加的三个具体系统名称
- [ ] 不评估这些系统的设计是否优秀 — 仅评估是否符合计划

### 用例 4：冲突升级 — 正确的上级
**场景：** game-designer 希望在临近截止前增加一个机制（影响所有游戏系统的动态天气），technical-director 警告这将需要额外 3 个冲刺。game-designer 和 technical-director 在是否推进方面存在分歧。
**预期：** Producer 不对该机制是否值得添加（设计决策）或是否可行（技术决策）表态。Producer 量化生产影响（3 个冲刺延迟、里程碑推迟风险），将权衡方案呈现给用户，并遵循 coordination-rules.md 冲突解决流程：升级至共同上级（本例中，由于 creative-director 和 technical-director 均为最高层，应将冲突呈现给用户决策）。
**断言：**
- [ ] 以具体数据（冲刺数、里程碑日期推迟）量化生产影响
- [ ] 不做约束性设计或技术决策
- [ ] 向用户呈现冲突，并清晰说明范围影响
- [ ] 引用 coordination-rules.md 冲突解决协议（升级至共同上级或用户）

### 用例 5：上下文传递 — 使用所提供的上下文
**场景：** Agent 收到一个关卡上下文块，包含当前里程碑截止日期（8 周后）和过去 4 个冲刺的速度数据（8、10、9、11 个故事点）。提交了一个包含 14 个故事点的冲刺计划。
**预期：** 评估使用所提供的速度数据来预测 14 个故事点是否可达，并引用 8 周的里程碑窗口来评估当前冲刺范围是否留有足够缓冲。
**断言：**
- [ ] 使用所提供上下文中的具体速度数字（而非通用估算）
- [ ] 在容量评估中引用 8 周的截止期限
- [ ] 计算或估算里程碑窗口内的剩余冲刺数
- [ ] 不给出与所提供截止日期和速度数据脱节的通用范围建议

## 协议合规性

- [ ] 仅使用 REALISTIC / CONCERNS / UNREALISTIC 词汇返回裁决
- [ ] 严守声明的生产领域
- [ ] 通过量化范围影响并呈现给用户来升级设计/技术冲突
- [ ] 在输出中使用关卡 ID（例如 `PR-SPRINT: REALISTIC`），而非内联散文裁决
- [ ] 不做约束性游戏设计或技术架构决策
