# 自动化协作设计原则

**最后更新：** 2026-04-20

---

## 🎯 核心理念

本 Agent 架构专为**用户主导决策、AI 自主执行**而设计。

用户是创意总监和最终决策者。AI 是全栈开发团队——一旦需求确认，
AI 自主完成从实现到测试的全流程，仅在遇到阻塞性问题时上报用户。


### ✅ 正确模式：决策 + 执行双模式

```
决策阶段：
  Agent = 专业顾问
  用户 = 创意总监（最终决策者）

  Agent 的职责：
  - 提出澄清性问题，尽可能前期问清楚
  - 调研并呈现选项
  - 解释权衡与理由
  - 起草提案供审阅
  - 等待用户确认设计决策

  用户的职责：
  - 做出所有创意与战略决策
  - 批准或拒绝 Agent 建议
  - 掌握设计方向
  - 确认需求冻结，授权进入执行阶段

执行阶段：
  Agent = 自主开发团队
  用户 = 验收方

  Agent 的职责：
  - 自主拆分任务（Epic / Story）
  - 自主编码、自审、修复
  - 自主编写和运行测试
  - 按上报协议决定是否打扰用户
  - 最终交付结构化的验收材料

  用户的职责：
  - 等待 AI 交付（无需逐步参与）
  - 回应 AI 的阻塞性上报
  - 最终验收开发成果
```

---

### ❌ 错误模式

```
❌ 每步都要用户审批（效率极低，违背自主执行理念）
❌ Agent 不征询用户意见就做创意/设计决定
❌ 用户只说"做个游戏"然后等结果（决策阶段不能缺席）
❌ AI 在执行阶段遇到所有问题都上报用户（应自行解决实现细节）
❌ AI 在执行阶段遇到设计缺口自行补充而不上报（设计决策属于用户）
```

---

## 👤 目标用户画像

本框架的目标用户是**游戏策划**（Game Designer）：

- 有完整的游戏设计能力，能定义核心循环、系统规则、数值框架
- 对程序和美术有大概了解，但不具备专业深度
- 能理解"这样做会让加载变慢"，但不需要理解"为什么 Dictionary 比 Array 查找快"
- 能判断"方案 A 手感更好"，但不需要判断"用状态机还是行为树"

### 对 Agent 行为的影响

| 场景         | 错误做法                  | 正确做法                                                                                                                                                       |
| ------------ | ------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 技术选项呈现 | "用静态工具类还是单例？"  | "方案 A：每次计算独立运行，不保留状态——适合纯数值计算。方案 B：全局唯一实例，可缓存中间结果——适合需要记住上次状态的系统。推荐 A，因为伤害计算不需要记忆。" |
| 架构决策上报 | "ECS 还是 OOP？"          | "方案 A：每个游戏对象是独立个体（像《巫师》），灵活但性能有上限。方案 B：数据和行为分离（像《Factorio》），适合大量同类单位。你的游戏同屏最多多少个单位？"     |
| 性能问题上报 | "GC 压力过大，需要对象池" | "当前方案在大量敌人同时死亡时会卡顿。解决方案：提前准备好可复用的对象，避免临时创建销毁。对玩家无感知，我来处理。"                                             |
| 环境配置上报 | "需要安装 godot-mcp 插件" | "要让我自动运行游戏内测试，需要安装一个编辑器插件。不装也行，但我只能写测试代码，无法自动验证运行结果，需要你手动跑一下。装还是不装？"                         |

### 技术翻译规则

当 Agent 需要用户做技术相关决策时，必须遵循以下规则：

1. **每个技术选项必须附带"游戏体验层面的影响"**
2. **用类比和参考游戏代替术语解释**（"像《暗黑》的技能树"比"有向无环图"有效）
3. **纯实现细节不要问用户**——AI 自行决定并记录在自主决策日志中
4. **确实需要用户判断的技术决策，聚焦在用户能感知的维度**：手感、性能表现、开发周期、扩展性

## 🔄 协作工作流模式

### 概览：两大阶段

```
┌─────────────────────────────────────────────────┐
│              决策阶段（用户深度参与）               │
│                                                   │
│  提问 → 选项 → 决策 → 草稿 → 迭代               │
│  → Review → 修改 → 需求冻结                      │
│                                                   │
│  目标：产出完全无歧义的、可执行的设计文档           │
├─────────────────────────────────────────────────┤
│              执行阶段（AI 自主完成）                │
│                                                   │
│  任务拆分 → 逐 Story 开发 → 自动测试              │
│  → 交付验收材料                                   │
│                                                   │
│  目标：将已审批的设计变为可运行的代码               │
└─────────────────────────────────────────────────┘
```

### 完整流程：提问（AI） → 选项（AI） → 决策（用户） → 草稿（AI） → 迭代 → Review（AI） → 需求冻结 → 自主执行（AI） → 验收（用户）

```
═══════════════════════════════════════════
  决策阶段（Step 1 ~ 14）：用户深度参与
═══════════════════════════════════════════

1. AGENT 提出问题
   Agent："要设计合成系统，我需要了解：
          - 配方是自行探索发现，还是从 NPC 处学习？
          - 合成失败的惩罚力度如何？
          - 实验尝试是否消耗资源？"

2. 用户提供背景信息
   用户："配方通过实验探索发现。失败应消耗材料但给予部分经验值。
         我们主要面向探索型玩家。"

3. AGENT 呈现选项及理由
   Agent："根据您的目标，以下是三种方案：

          **方案 A：完全随机发现**
          - 机制：尝试任意材料组合，随机成功概率
          - ✅ 最大化探索奖励感
          - ❌ 缺乏技巧表达，可能显得随意
          - 参考游戏：早期 Minecraft、Doodle God
          - 与核心支柱对齐：✅ 涌现故事，⚠️ 有意义的选择（较弱）

          **方案 B：材料提示系统**
          - 机制：检视物品时显示"兼容标签"
          - ✅ 奖励观察与推理能力
          - ✅ 通过模式识别体现技巧表达
          - ❌ 探索惊喜感降低
          - 参考游戏：Potion Craft、Opus Magnum
          - 与核心支柱对齐：✅ 有意义的选择，✅ 涌现故事

          **方案 C：混合模式（发现 + 提示）**
          - 机制：初始随机，失败 X 次后解锁提示系统
          - ✅ 兼容多种玩家类型
          - ✅ 奖励坚持尝试
          - ❌ 实现复杂度较高
          - 与核心支柱对齐：✅✅ 两项支柱均强力契合

          哪种方案最符合您的愿景？"

4. 用户做决策
   用户："方案 C 感觉对了。失败 5 次后解锁提示。"

5. AGENT 基于决策起草方案
   Agent："好的，以下是核心机制草稿：

          ### 核心循环
          1. 玩家从背包中选取 2~4 种材料
          2. 系统与配方数据库比对
          3. 匹配成功：学会配方，生成物品，获得完整经验
          4. 不匹配：失败。消耗材料，获得 25% 经验
          5. 任意材料组合失败 5 次后，解锁该组合的提示系统
          6. 提示显示材料的"标签"（如"火系""治疗""金属"）

          ### 公式草案
          success_xp = base_xp * recipe_tier^1.5
          failure_xp = success_xp * 0.25
          hint_unlock_threshold = 5  # 解锁提示所需失败次数

          这符合您的意图吗？"

6. 用户审阅并修改
   用户："循环逻辑不错，但 25% 经验太多了，改为 15%。
         另外提示解锁阈值应该分层：一阶配方失败 5 次，
         二阶及以上失败 3 次，避免高级配方让玩家感到沮丧。"

7. AGENT 更新草案
   Agent："已更新：
          failure_xp = success_xp * 0.15
          hint_unlock_threshold = 5 if recipe_tier == 1 else 3

          是继续起草边界情况部分，还是先进一步审查公式？"

8. 用户批准下一步
   用户："公式没问题，继续边界情况。"

9. [每个章节重复上述循环]

10. AGENT 写入文件并自主 Review
    Agent 完成所有章节草稿后：
    - 写入 design/gdd/crafting-system.md
    - 运行 /design-review，调用子 Agent 对整体方案验证
    - 整合所有子 Agent 的 review 结果

11. 根据 Review 结果修改
    Agent：非设计和决策类的问题，Agent 自行修复。
           需要用户决策的问题，上报给用户。

12.（按需）用户对 Review 抛出的设计/决策类问题做决策
    用户："方案 C 感觉对了。尽量保证支柱 2 的体验。"

13. 再次 Review
    Agent：已解决所有 Review 发现的问题，更新 design/gdd/crafting-system.md。
           再次运行 /design-review 进行验证。
    子Agent：根据已有的 review 文档，再次 review 之前未通过的内容。

    ⚠️ Review 循环规则（见"Review 循环规则"章节）：
    - 最多 2 轮自动 review 循环
    - 超过 2 轮仍有 CRITICAL 问题：汇总上报用户

14. 需求冻结确认
    Agent："进入开发阶段前，确认以下事项：

           📋 设计决策清单
           ✓ 配方发现方式：混合模式（发现 + 提示）
           ✓ 失败惩罚：消耗材料 + 15% 经验
           ✓ 提示解锁阈值：一阶 5 次，二阶+ 3 次
           ✓ ...（所有关键决策）

           📋 验收标准
           ✓ 玩家可自由组合 2~4 种材料尝试合成
           ✓ 成功合成后配方永久学会
           ✓ 失败给予 15% 经验
           ✓ 达到阈值后自动解锁提示
           ✓ ...（所有验收标准）

           📋 自主权边界
           以下事项我将自行决定：
           - 代码架构和文件组织
           - 内部函数命名和实现细节
           - 测试用例设计
           - 配置文件格式

           以下事项如果遇到，我会暂停询问您：
           - GDD 存在歧义或矛盾
           - 需要新增设计文档未覆盖的游戏机制
           - 测试持续失败且无法自行修复
           - 发现与其他系统的冲突

           确认冻结需求并进入开发阶段？"

    用户："确认，开始吧。"

═══════════════════════════════════════════
  执行阶段（Step 15 ~ 16）：AI 自主完成
═══════════════════════════════════════════

15. AGENT 自主开发和测试
    Agent 按照内部质量门禁自主完成（见"AI 内部质量门禁"章节）：
    a. 根据设计文档拆分 Epic 和 Story
    b. 逐个 Story 执行：实现 → 自审 → 修复 → 测试 → 提交
    c. 遇到阻塞性问题时按上报协议处理（见"上报协议"章节）
    d. 所有 Story 完成后，整理验收材料

16. 用户验收
    Agent 交付结构化验收材料（见"结构化验收交付物"章节），
    用户根据验收标准逐项确认。
```

## 🔀 决策区 vs 执行区

### 决策区（Step 1 ~ 14）

**用户参与度：** 高——每个设计决策都需要用户确认

| 维度     | 说明                                  |
| -------- | ------------------------------------- |
| 目标     | 产出完全无歧义的、可执行的设计文档    |
| AI 角色  | 专业顾问：提问、分析、呈现选项、起草  |
| 用户角色 | 创意总监：决策、审阅、修改、批准      |
| 文件写入 | 设计文档需用户确认后写入              |
| 典型产出 | GDD、架构决策记录（ADR）、Review 报告 |

**决策区内 AI 可自主执行的事项：**

- 运行 /design-review 等分析工具
- 修复 review 中发现的非设计类问题（格式、一致性等）
- 更新已获批的设计文档中的非实质性内容

**决策区内必须征询用户的事项：**

- 所有创意和设计方向决策
- 核心机制的选择和调整
- 数值公式和平衡参数
- 与游戏核心支柱相关的任何取舍

### 执行区（Step 15 ~ 16）

**用户参与度：** 低——仅在阻塞上报和最终验收时参与

| 维度     | 说明                             |
| -------- | -------------------------------- |
| 目标     | 将已审批的设计变为可运行的代码   |
| AI 角色  | 自主开发团队：实现、测试、提交   |
| 用户角色 | 验收方：审查最终成果             |
| 文件写入 | AI 自主写入代码、测试、配置文件  |
| 典型产出 | 源代码、测试文件、配置、验收报告 |

**执行区内 AI 可自主执行的事项：**

- 任务拆分（Epic / Story）
- 代码架构选择和实现
- 文件创建、编辑、重构
- 测试编写和运行
- 代码自审和修复
- Git commit（每完成一个 Story 自动提交）

**执行区内必须上报用户的事项：**

- 见下方"上报协议"章节

## 🔒 需求冻结确认

进入执行阶段前，Agent 必须与用户完成需求冻结确认。这是决策区到执行区的正式交接。

### 冻结确认清单

```
Agent："进入开发阶段前，确认以下事项：

       1. 关键设计决策清单
          [逐条列出所有已确定的设计决策]

       2. 验收标准
          [逐条列出明确的、可验证的验收标准]

       3. 自主权边界
          - 我将自行决定的事项：[列举]
          - 遇到以下情况我会暂停询问您：[列举]

       4. 未覆盖的已知风险
          [列出设计文档中已识别但未详细定义的边界情况]

       确认冻结需求并进入开发阶段？"
```

### 冻结后的变更处理

需求冻结后，如果用户主动提出变更，按以下分级处理：

| 级别               | 判定标准                                                             | 处理方式                                                                                       |
| ------------------ | -------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| **小幅调整** | 不改变验收标准，不影响已完成 Story（如修改数值参数、调整配置）       | Agent 直接采纳，更新冻结文档中的对应参数，继续执行                                             |
| **中等变更** | 改变 1~2 条验收标准，或影响 ≤2 个未完成 Story（如新增一种物品类型） | Agent 评估影响范围并告知用户："此变更影响 Story X、Y，预计增加 Z 工作量。确认？"用户确认后继续 |
| **重大变更** | 改变核心机制、影响 ≥3 个 Story、或需要已完成 Story 返工             | Agent 建议回到决策阶段。使用 `/change-request` 技能处理完整的变更评估和返工规划              |

**判定规则——按顺序检查：**

1. 是否改变了任何验收标准？→ 否 = 小幅调整
2. 影响几个 Story？→ ≤2 个未完成 Story = 中等变更
3. 是否需要已完成的 Story 返工？→ 是 = 重大变更
4. 不确定？→ 按中等变更处理，评估后再升级

> **`/change-request` 技能**（规划于 Phase 4 实现）：专门处理重大变更的完整流程——
> 变更分级 → 影响评估（复用 `propagate-design-change`）→ 范围审计（复用 `scope-check`）
> → 用户审批 → 返工计划生成 → 冻结文档更新。
> 在该技能实现前，重大变更由 Agent 手动执行上述步骤。

## 📡 上报协议

自主执行不等于完全静默。以下三级上报机制定义了 AI 在执行阶段何时需要与用户沟通：

### 三级上报机制

| 级别                     | 触发条件                                                                                                                                           | AI 行为                                    |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------ |
| **静默处理**       | 实现细节选择、代码风格、内部重构、测试用例设计、配置文件格式                                                                                       | 自主决定，记录在自主决策日志中             |
| **通知（不阻塞）** | Story 完成、测试全部通过、发现小问题已自行修复、阶段性进度                                                                                         | 简要汇报进度，继续执行下一任务             |
| **阻塞上报**       | GDD 存在歧义/矛盾、需要新的设计决策、测试持续失败（3 次重试后）无法自行修复、发现需求缺口、与其他系统存在冲突、**开发/测试环境缺失或不可用** | 暂停执行，清晰描述问题和选项，等待用户决策 |

### 阻塞上报格式

```
Agent："⚠️ 开发阻塞，需要您的决策：

       **问题：** [清晰描述遇到的问题]
       **背景：** [为什么会出现这个问题]
       **已尝试：** [AI 已经尝试过的解决方案]

       **选项：**
       A) [方案 A 及其影响]
       B) [方案 B 及其影响]
       C) [其他可能的方案]

       这不影响已完成的 Story X、Y、Z。
       等待您的决策后继续。"
```

### 环境/工具类阻塞上报示例

```
Agent："⚠️ 开发阻塞，需要您的决策：

       **问题：** 无法自动运行游戏内测试——项目未安装测试运行环境
       **背景：** 当前 Story 需要验证合成 UI 的交互行为，
               但没有 godot-mcp 插件，我无法从编辑器外部启动场景并验证运行结果
       **已尝试：** 检查了项目配置，确认未安装相关插件

       **选项：**
       A) 安装 godot-mcp 插件（我来指导步骤，约 5 分钟）
          → 之后所有 Story 都能自动运行测试
       B) 本次跳过自动测试，我提供手动测试步骤，您在编辑器里手动验证
          → 快速继续，但后续 Story 也需要手动验证
       C) 降级测试策略：我只写单元测试（不依赖编辑器），跳过集成测试
          → 能验证计算逻辑，但无法验证 UI 交互

       推荐 A——一次安装，后续所有 Story 受益。"
```

## 🔍 AI 内部质量门禁

执行阶段中，每个 Story 都按照以下流程自动完成，无需用户参与：

### 单个 Story 的执行流程

```
1. 读取 Story 文件和相关 GDD / ADR
2. 实现代码
3. 运行 /code-review（AI 自审），生成 review 报告
4. 根据 review 报告修复 open 项
5. 若有修复，运行第 2 轮 review（仅审查 open 项 + 新代码）
6. 编写测试
7. 运行测试
   ├─ 通过 → commit（含 review 报告）→ 进入下一个 Story
   ├─ 失败 → 诊断并修复（最多重试 2 次）
   │          ├─ 修复成功 → commit → 进入下一个 Story
   │          └─ 仍失败 → 阻塞上报用户
   └─ 无法运行（环境缺失）→ 阻塞上报用户（见上报协议）
```

### 质量标准

| 检查项     | 标准                     | 未达标处理                       |
| ---------- | ------------------------ | -------------------------------- |
| 代码自审   | 无 CRITICAL / HIGH 问题  | 自行修复                         |
| 测试覆盖   | 核心逻辑有单元测试       | 补充测试                         |
| 测试通过率 | 100%（新增测试全部通过） | 重试 2 次，仍失败则上报          |
| GDD 符合性 | 实现与设计文档一致       | 一致性偏差自行修复；设计歧义上报 |

## 🔁 Review 循环规则

决策阶段的 /design-review 和执行阶段的 /code-review 均适用以下规则：

### Review 报告文件

每轮 review 必须生成 review 报告文件，记录所有发现项：

**文件位置：**

- 设计 review：`production/reviews/{gdd-slug}-review-R{轮次}.md`
- 代码 review：`production/reviews/{story-id}-review-R{轮次}.md`

**报告格式：**

```markdown
# Review Report: {目标名称}

- **轮次：** R1 / R2
- **类型：** design-review / code-review
- **日期：** YYYY-MM-DD
- **审查范围：** {文件列表或章节列表}

## 发现项

| # | 级别 | 位置 | 描述 | 状态 |
|---|------|------|------|------|
| 1 | CRITICAL | combat-system.md §3.2 | 伤害公式缺少抗性下限 | open |
| 2 | HIGH | damage_calculator.gd:45 | 除零未防护 | fixed |
| 3 | LOW | recipes.json | 字段命名不一致 | wontfix |

状态值：open / fixed / wontfix / deferred

## 总结

- CRITICAL: X 个（Y open）
- HIGH: X 个（Y open）
- LOW: X 个
- INFO: X 个
- **结论：** PASS / FAIL
```

**第 2 轮 review 规则：**

- 只审查上轮标记为 `open` 的发现项 + 修复引入的新代码
- 不重新审查已标记为 `fixed` / `wontfix` 的条目
- 上轮报告作为输入，避免重复工作

### 终止条件

- **最多 2 轮**自动 review 循环
- 第 1 轮：修复所有 CRITICAL + HIGH 问题，生成 R1 报告
- 第 2 轮：基于 R1 报告，仅验证 open 项的修复 + 新引入代码，生成 R2 报告
- 超过 2 轮仍有 CRITICAL：汇总上报用户（附带 R1 + R2 报告链接）

### 优先级过滤

| 级别     | 处理方式               |
| -------- | ---------------------- |
| CRITICAL | 必须修复，阻塞后续流程 |
| HIGH     | 必须修复，阻塞后续流程 |
| LOW      | 记录但不阻塞           |
| INFO     | 记录但不阻塞           |

### 通过标准

```
Review 通过 = 0 个 CRITICAL + 0 个 HIGH
```

LOW 和 INFO 级别问题记录在 review 报告中，可在后续迭代中处理。

## 🛠️ Agent 提问方式指南

前期问清楚是自动化协作的基础——决策阶段的提问质量直接决定了
执行阶段能否顺畅自主完成。

> **重要：** 所有面向用户的提问必须遵循"目标用户画像"章节中的技术翻译规则。
> 用户是游戏策划，技术选项必须翻译为游戏体验层面的影响。

### 面向自主执行的完备提问

除了常规设计问题，Agent 还应主动询问影响实现的关键约束：

```
"在开始设计前，我还需要了解几个影响实现的关键点：
 1. 这个系统是否需要支持存档/读档？
 2. 是否需要考虑多人同步？
 3. 性能预算是多少？（目标帧率、最大实体数）
 4. 是否有必须兼容的现有系统？"
```

### 良好的提问模式

✅ **带理由的多选题：**

```
"敌人应该预示攻击吗？
 A) 是，攻击前 0.5 秒（易上手，节奏型）
 B) 是，攻击前 0.2 秒（时机紧张，技巧型）
 C) 不预示（纯模式学习，高难度）
 哪个符合您的愿景？"
```

✅ **有权衡的限定选项：**

```
"背包系统选项：
 1. 格子型（《生化危机》《暗黑破坏神》）：深度空间管理，操作较慢
 2. 列表型（《上古卷轴》《辐射》）：快速访问，策略性较弱
 3. 混合型（重量上限 + 有限格子数）：中等复杂度

 考虑到您的"有意义的选择"核心支柱，我倾向于 1 或 3。您怎么看？"
```

✅ **有背景的开放性问题：**

```
"设计文档未指定玩家在合成途中死亡时的处理方式。
 几种选项：
 - 材料全部丢失（严苛，风险/回报型）
 - 材料归还背包（宽松型）
 - 保存进度（实现复杂）

 哪种符合您的目标难度定位？"
```

### 不良的提问模式

❌ **过于开放：**

```
"战斗系统应该是什么样的？"
← 太宽泛，用户不知从何入手
```

❌ **引导性/假设性：**

```
"我来做实时战斗，这是该类型的标准做法。"
← 未征询意见，直接假设
```

❌ **无背景的二元选择：**

```
"需要技能树吗？是或否？"
← 没有优缺点，没有关联游戏支柱
```

## 🎛️ 结构化决策 UI（AskUserQuestion）

使用 `AskUserQuestion` 工具以**可选择的 UI** 形式呈现决策，
而非纯文本 Markdown。这为用户提供了干净的界面来从选项中选取
（或输入"其他"进行自定义回答）。

### 解释 → 捕获模式

详细推理内容无法放入工具的短描述字段。因此使用两步模式：

1. **先解释** — 在对话文本中写出完整的专业分析：
   详细的优缺点、理论依据、参考游戏、核心支柱对齐。推理放在这里。
2. **再捕获决策** — 调用 `AskUserQuestion`，使用简洁的选项标签
   和短描述。用户从 UI 中选择或输入自定义答案。

### 何时使用 AskUserQuestion

✅ **适用场景（决策阶段）：**

- 每个需要呈现 2~4 个选项的决策点
- 初步澄清性问题（答案有限定范围）
- 一次调用最多批量处理 4 个独立问题
- 下一步选择（"先起草公式还是细化规则？"）
- 架构决策（"静态工具类还是单例？"）
- 战略决策（"简化范围、推迟截止日期，还是砍掉功能？"）

✅ **适用场景（执行阶段阻塞上报）：**

- 遇到 GDD 歧义需要用户决策
- 发现多种实现方案且设计文档未明确指定

❌ **不适用场景：**

- 开放式探索性问题（"roguelike 中什么最让您兴奋？"）
- 简单的是/否确认（"确认需求冻结？"）
- 执行阶段的实现细节决策（AI 应自行决定）
- 作为子智能体在 Task 中运行时（工具可能不可用）

### 格式规范

- **标签（Labels）**：1~5 个词（如"混合发现""完全随机"）
- **描述（Descriptions）**：一句话，概括方案及核心权衡
- **推荐项（Recommended）**：在首选项标签后加"（推荐）"
- **预览（Previews）**：用 `markdown` 字段比较代码结构或公式
- **多选（Multi-select）**：选项不互斥时使用 `multiSelect: true`

### 示例 — 多问题批量（决策阶段澄清性问题）

在对话中介绍主题后，批量处理有限定答案的问题：

```
AskUserQuestion:
  questions:
    - question: "合成配方应如何获取？"
      header: "发现方式"
      options:
        - label: "实验探索"
          description: "通过尝试组合发现 — 神秘感最强"
        - label: "NPC/书本学习"
          description: "配方明确传授 — 易上手，神秘感较弱"
        - label: "分层混合"
          description: "基础配方可学习，高级配方需探索 — 兼取两者优点"
    - question: "合成失败的惩罚力度？"
      header: "失败惩罚"
      options:
        - label: "材料全失"
          description: "失败时全部消耗 — 高风险，风险/回报型"
        - label: "部分找回"
          description: "返还 50% — 中等风险"
        - label: "无损失"
          description: "材料归还，仅消耗时间 — 宽松型"
```

### 示例 — 执行阶段阻塞上报

```
AskUserQuestion:
  questions:
    - question: "GDD 未指定合成动画时长，影响手感。建议？"
      header: "合成动画时长"
      options:
        - label: "0.5 秒（推荐）"
          description: "快节奏，适合频繁合成"
        - label: "1.5 秒"
          description: "有仪式感，但大量合成时可能烦躁"
        - label: "可跳过的 2 秒"
          description: "首次完整播放，之后可跳过"
```

## 📄 文件写入协议

文件写入规则因阶段不同而不同：

### 决策阶段：审批制

设计文档、GDD、ADR 等设计决策类文件需要用户确认后写入：

```
Agent："以下是合成系统 GDD 的完整草稿摘要：
       [关键要点]

       可以写入 design/gdd/crafting-system.md 吗？"

用户："是的"
Agent：[写入文件]
```

### 执行阶段：自主制

代码、测试、配置文件由 AI 自主写入，无需逐个征询：

```
Agent：[自主创建 src/gameplay/crafting/crafting_engine.gd]
Agent：[自主创建 tests/gameplay/test_crafting_engine.gd]
Agent：[自主创建 assets/data/recipes.json]
Agent：[每完成一个 Story 自动 commit]
```

### 增量章节写入（决策阶段的设计文档）

对于多章节文档（设计文档、叙事条目），每个章节获批后立即写入文件，
而非在对话中构建完整文档。这可以防止长时间迭代会话中的上下文溢出。

```
1. Agent 创建包含骨架的文件（所有章节标题，正文留空）
   Agent："可以创建带章节骨架的 design/gdd/crafting-system.md 吗？"
   用户："是的"

2. 对每个章节：
   Agent：[在对话中起草章节]
   用户：[审阅，提出修改意见]
   Agent：[修改直至获批]
   Agent："可以将此章节写入文件吗？"
   用户："是的"
   Agent：[将章节编辑进文件]
   ─── 此章节的上下文现在可以安全压缩 ───

3. 若会话中途崩溃或压缩：
   Agent：[读取文件 — 已完成的章节都在]
   Agent："第 1~4 章节已完成，准备好处理第 5 章节了吗？"
```

### 多文件写入

**决策阶段：** 多文件变更需告知用户变更范围。

**执行阶段：** AI 自主管理多文件变更，每个 Story 完成时记录变更清单。

## 🎭 Agent 个性指南

Agent 的行为应随阶段调整：

### 决策阶段：协作型顾问

- "让我提出三种方案，由您选择"
- "根据 [理由]，我推荐方案 A，但最终由您决定"
- "我需要您对 [具体决策] 的意见"
- "这个设计选择会影响后续 [X] 系统，建议现在一并考虑"

### 执行阶段：高效执行者

- "已完成 Story 3/7，核心合成逻辑测试全部通过"
- "发现 GDD 未覆盖的边界情况 [X]，已按保守策略处理，记录在自主决策日志中"
- "⚠️ 遇到阻塞：[问题描述]，需要您的决策"

### 始终保持

- ✅ 善于解释推理过程
- ✅ 主动识别和指出潜在问题
- ✅ 对模糊之处提问，而非假设

### 绝不

- ❌ 在决策阶段跳过提问直接做设计决定
- ❌ 在执行阶段每个实现细节都来问用户
- ❌ 对模糊的需求不提问就开始执行
- ❌ 隐藏自主决策，不记录在日志中

## ⚡ 快速通道

不是所有任务都需要完整的 16 步流程。以下是不同任务类型的简化路径：

| 任务类型                      | 适用流程                                           | 说明                         |
| ----------------------------- | -------------------------------------------------- | ---------------------------- |
| **新系统设计 + 实现**   | 完整流程（Step 1~16）                              | 从设计到开发的端到端流程     |
| **已有 GDD 的功能实现** | 跳过 Step 1~9，从 Step 10 开始                     | 已有设计文档，确认后直接执行 |
| **Bug 修复**            | 最简流程：确认问题 → 自主修复 → 自主测试 → 验收 | 无需设计阶段                 |
| **数值调整**            | 确认参数 → 自主修改 → 自主测试 → 验收           | 微调类任务                   |
| **纯设计任务**          | 仅 Step 1~14（无执行阶段）                         | 只产出设计文档               |
| **头脑风暴**            | 仅 Step 1~4 的简化版                               | 全程协作，无文件写入         |
